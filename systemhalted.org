* Overview

This is a compact Emacs configuration focused on Org mode, Java, Python and
Jupyter notebooks, web programming, Git, and documentation formats.

The guiding rule is simple: keep Emacs excellent at the work that happens here
every day, and remove stacks that mainly preserve old habits. 

The current direction favors composable packages that use Emacs' built-in
completion and language APIs. Large, all-in-one frameworks were replaced where a
smaller stack gives the same workflow with less hidden state. Org is treated as a
system for thinking, consuming, and executing, not as a maximal task manager.

* Startup and Package Management

Startup is intentionally boring. Package archives are current and explicit:
GNU/Nongnu provide core ecosystem packages, MELPA provides actively maintained
tooling, and MELPA Stable remains lower priority. =use-package= is used only to
make package ownership and configuration boundaries obvious. External packages
declare =:ensure t= at the call site; built-in packages declare =:ensure nil=.
There is no global auto-install switch because broad implicit installs make
reloads harder to reason about.

Emacs Custom can still write local state, but it is redirected under
=var/custom.el= so generated choices do not churn this repository. Settings that
are part of the intentional configuration should live here instead. Opaque
fingerprints, such as trusted theme hashes, can remain in ignored Custom state;
readable preferences, such as the =lsp-ui= face overrides below, belong in the
literate config.

Native compilation runs asynchronously for installed packages, and several
upstream packages (=dap-mode=, =lsp-java=, =posframe=, =lsp-treemacs=) emit
"function not known to be defined" warnings because cross-package references are
visible to the byte compiler before their =declare-function= forms load. The
warnings are benign — every symbol resolves at runtime via autoloads — so the
Warnings buffer is silenced. Messages are still written to the
=*Native-compile-Log*= buffer if a real failure ever needs investigation.

#+begin_src emacs-lisp
;;; systemhalted.el --- Personal Emacs configuration -*- lexical-binding: t; -*-

(setq gc-cons-threshold (* 50 1000 1000))

(setq native-comp-async-report-warnings-errors 'silent)

(add-hook 'emacs-startup-hook
          (lambda ()
            (message "Emacs loaded in %.2f seconds with %d garbage collections."
                     (float-time (time-subtract after-init-time before-init-time))
                     gcs-done)))

(setq custom-file (expand-file-name "var/custom.el" user-emacs-directory))
(make-directory (file-name-directory custom-file) t)
(when (file-exists-p custom-file)
  (load custom-file))

(require 'package)
(setq package-enable-at-startup nil
      package-archives '(("gnu" . "https://elpa.gnu.org/packages/")
                         ("nongnu" . "https://elpa.nongnu.org/nongnu/")
                         ("melpa" . "https://melpa.org/packages/")
                         ("melpa-stable" . "https://stable.melpa.org/packages/"))
      package-archive-priorities '(("gnu" . 30)
                                   ("nongnu" . 25)
                                   ("melpa" . 20)
                                   ("melpa-stable" . 10)))
(package-initialize)
(unless package-archive-contents
  (package-refresh-contents))

(unless (package-installed-p 'use-package)
  (package-install 'use-package))
(require 'use-package)
(setq use-package-always-ensure nil)
#+end_src

* Core Editing

The editor baseline before any package loads: quiet startup, runtime files
routed under =~/.emacs.d/=, fullscreen frames, line-number safety, an
emergency UI reset for runaway sessions, and the commands that make this
literate file easy to edit from inside Emacs.

** Editor defaults and runtime files

One =setq= block establishes the boring baseline; the clusters it covers:

- *Quiet startup.* =inhibit-startup-screen=, =initial-scratch-message nil=,
  and =initial-major-mode 'org-mode= replace the splash screen with an empty
  =*scratch*= in Org mode — this file's native format.
- *Sensory quiet.* =ring-bell-function 'ignore= silences the system beep;
  =echo-keystrokes 0.01= makes the minibuffer show the in-progress chord
  almost immediately so which-key has something to attach to.
- *Confirm kill.* =confirm-kill-emacs= guards =C-x C-c= so a stray chord
  can't take down the session.
- *Backups routed locally.* =backup-directory-alist= sends every =~filename=
  to =~/.emacs.d/backups/= and =auto-save-file-name-transforms= sends
  =#filename#= to =~/.emacs.d/auto-save-list/=. The repository's
  =.gitignore= excludes both, keeping the working tree clean.
- *Lock files off.* =create-lockfiles nil= drops the =.#filename= symlinks
  Emacs writes next to a buffer on first edit. This is a single-user setup
  where the concurrent-edit warning effectively never fires, and the
  symlinks confuse external watchers (bundlers, file-system watchers,
  language servers, Git GUIs).
- *Versioned backups.* =backup-by-copying=, =delete-old-versions=, and the
  =kept-new=/=kept-old= caps keep a small rolling history per file.

After the =setq=, =y-or-n-p= replaces =yes-or-no-p= globally (one-key
confirmation), and UTF-8 is asserted across coding system, language
environment, and default I/O.

#+begin_src emacs-lisp
  (setq inhibit-startup-screen t
        initial-scratch-message nil
        initial-major-mode 'org-mode
        ring-bell-function 'ignore
        echo-keystrokes 0.01
        confirm-kill-emacs #'yes-or-no-p
        backup-by-copying t
        backup-directory-alist `(("." . ,(expand-file-name "backups/" user-emacs-directory)))
        auto-save-file-name-transforms `((".*" ,(expand-file-name "auto-save-list/" user-emacs-directory) t))
        create-lockfiles nil
        delete-old-versions t
        kept-new-versions 3
        kept-old-versions 2
        version-control t)

  (fset 'yes-or-no-p #'y-or-n-p)
  (prefer-coding-system 'utf-8)
  (set-language-environment 'utf-8)
  (set-default-coding-systems 'utf-8)
#+end_src

** Customization group and chrome

=defgroup systemhalted= namespaces every =defcustom= in this config so they
all surface together under =M-x customize-group RET systemhalted=. The three
=…-mode -1= calls strip the menu bar, tool bar, and scroll bars before any

    :group 'convenience)

** Fullscreen frames

Both the initial frame and any subsequent frames open fullscreen. The
portable bit is =(fullscreen . fullboth)= in the frame alists, which works
under X11 and PGTK. =ns-use-native-fullscreen= is the macOS-only toggle that
makes Cmd-Ctrl-F use the system's native fullscreen animation rather than a
maximized-but-windowed state; setting it is wrapped in a darwin check
because the variable is unbound on Linux.

#+begin_src emacs-lisp
  (when (display-graphic-p)
    ;; ns-use-native-fullscreen is the NeXTSTEP (macOS) fullscreen mode
    ;; toggle and is unbound on Linux; (fullscreen . fullboth) below is
    ;; the portable bit that handles X11/PGTK.
    (when (eq system-type 'darwin)
      (setq ns-use-native-fullscreen t)))
  (add-to-list 'initial-frame-alist '(fullscreen . fullboth))
  (add-to-list 'default-frame-alist '(fullscreen . fullboth))
#+end_src

** Line numbers, gated to safe buffers

Long-running Emacs sessions can become memory-bound when redisplay is forced
to combine expensive features in large or visually complex buffers. The
=display-line-numbers-mode= margin is one of those features. Rather than
turning it on globally, an eligibility predicate filters down to ordinary
local programming buffers:

- Mode must derive from =prog-mode= — text and Org buffers don't get the
  margin.
- Must be visiting a local file (not TRAMP, not =*scratch*=, not a generated
  Org export buffer like =systemhalted.html=).
- Buffer size below =systemhalted/line-number-max-buffer-size= (1 MiB) — at
  that point the margin's per-redisplay cost stops being free.
- No unusually long lines in the first 2 000 lines (=systemhalted/long-line-buffer-p=).
  Pathological long lines are what =so-long= is for, and the line-number
  gutter compounds their slowness.

=systemhalted/display-line-numbers-disable-all= is the explicit escape
hatch, also called from the emergency UI reset further down.

#+begin_src emacs-lisp
  (defcustom systemhalted/line-number-max-buffer-size (* 1024 1024)
    "Maximum buffer size where line numbers are enabled automatically."
    :type 'integer
    :group 'systemhalted)

  (defcustom systemhalted/line-number-max-line-length 1000
    "Maximum sampled line length where line numbers are enabled automatically."
    :type 'integer
    :group 'systemhalted)

  (defun systemhalted/long-line-buffer-p ()
    "Return non-nil when the current buffer has unusually long sampled lines."
    (save-excursion
      (save-restriction
        (widen)
        (goto-char (point-min))
        (let ((limit (save-excursion
                       (forward-line 2000)
                       (point)))
              (too-long nil))
          (while (and (not too-long) (< (point) limit))
            (let ((length (- (line-end-position) (line-beginning-position))))
              (when (> length systemhalted/line-number-max-line-length)
                (setq too-long t)))
            (forward-line 1))
          too-long))))

  (defun systemhalted/generated-export-buffer-p ()
    "Return non-nil when the current buffer visits a generated config export."
    (and buffer-file-name
         (member (file-name-nondirectory buffer-file-name)
                 '("systemhalted.md" "systemhalted.html" "systemhalted.tex"))))

  (defun systemhalted/display-line-numbers-eligible-p ()
    "Return non-nil when line numbers are useful and unlikely to hurt redisplay."
    (and (derived-mode-p 'prog-mode)
         buffer-file-name
         (not (file-remote-p buffer-file-name))
         (not (systemhalted/generated-export-buffer-p))
         (< (buffer-size) systemhalted/line-number-max-buffer-size)
         (not (systemhalted/long-line-buffer-p))
         (not (minibufferp))))

  (defun systemhalted/display-line-numbers-maybe ()
    "Enable line numbers only in normal local code buffers."
    (when (systemhalted/display-line-numbers-eligible-p)
      (display-line-numbers-mode 1)))

  (defun systemhalted/display-line-numbers-disable-all ()
    "Disable line numbers in every live buffer."
    (interactive)
    (dolist (buffer (buffer-list))
      (with-current-buffer buffer
        (when (bound-and-true-p display-line-numbers-mode)
          (display-line-numbers-mode -1)))))

  (add-hook 'prog-mode-hook #'systemhalted/display-line-numbers-maybe)
#+end_src

** Always-on minor modes and shared infrastructure

Eight short minor-mode toggles cover the everyday quality-of-life baseline:

- =column-number-mode= shows column position in the mode line; pairs with
  the default line count.
- =display-time-mode= / =display-battery-mode= add clock and battery
  indicators so a long-running Emacs session doesn't need a side glance.
- =electric-pair-mode= auto-inserts matching brackets and quotes in
  prog-mode; =show-paren-mode= highlights the matching delimiter under
  point.
- =save-place-mode= remembers point per file across restarts.
- =recentf-mode= keeps a recently-opened-files list that =consult-buffer=
  surfaces under the =f= narrowing key.
- =global-auto-revert-mode= refreshes buffers whose backing files changed
  on disk — important when language servers or formatters rewrite files
  underneath us.

The three =use-package= forms wire in built-ins the rest of the config
depends on: =seq= (used by helpers like =systemhalted/active-minor-modes=),
=so-long= (protects against pathological long lines that the line-number
gate above also accounts for), and =server= (starts the Emacs server when
not in batch mode, so =emacsclient= and the macOS =/Applications/Emacs.app=
icon attach to this process instead of spawning a new one).

#+begin_src emacs-lisp
  (column-number-mode 1)
  (display-time-mode 1)
  (display-battery-mode 1)
  (electric-pair-mode 1)
  (show-paren-mode 1)
  (save-place-mode 1)
  (recentf-mode 1)
  (global-auto-revert-mode 1)

  (use-package seq
    :ensure nil)

  (use-package so-long
    :ensure nil
    :config
    (global-so-long-mode 1))

  (use-package server
    :ensure nil
    :config
    (unless (or noninteractive (server-running-p))
      (server-start)))
#+end_src

** Diagnostics and the emergency UI reset

When the UI starts to degrade — sluggish redisplay, climbing RSS,
unresponsive minibuffer — these helpers tell you what's running and let
you shed the expensive bits without restarting Emacs.

=systemhalted/diagnostics= (=M-x= it) collects uptime, current buffer,
buffer size and longest-line sample, active minor modes, LSP state, live
timer counts, GC counter, and a memory summary into a single alist for
inspection or capture into a notes buffer.

=systemhalted/emergency-ui-reset= (bound to =C-c !=) is the actual
turn-it-off button. It disables line numbers everywhere, stops
=global-auto-revert-mode=, and walks every buffer killing Flycheck, LSP,
=mixed-pitch=, and =olivetti= modes where they are active; then it forces a
GC and prints the resulting memory summary. Cheap to invoke, completely
reversible — re-opening a file re-runs the relevant hooks.

If Emacs is unresponsive enough that =C-c != itself doesn't fire, sample
the process from a terminal (macOS):

#+begin_src sh
sample <pid> 5 -file /tmp/emacs-hang-sample.txt
#+end_src

On Linux the equivalent is =sudo perf record -F 99 -p <pid> -- sleep 5=
followed by =perf report=, or =py-spy dump --pid <pid>= for a one-shot
backtrace if =py-spy= happens to be installed.

#+begin_src emacs-lisp
  (defun systemhalted/buffer-longest-line-sample (&optional line-limit)
    "Return the longest line length found in the first LINE-LIMIT lines."
    (save-excursion
      (save-restriction
        (widen)
        (goto-char (point-min))
        (let ((remaining (or line-limit 2000))
              (longest 0))
          (while (and (> remaining 0) (not (eobp)))
            (setq longest (max longest (- (line-end-position) (line-beginning-position))))
            (setq remaining (1- remaining))
            (forward-line 1))
          longest))))

  (defun systemhalted/active-minor-modes ()
    "Return active minor modes for the current buffer."
    (seq-filter (lambda (mode)
                  (and (boundp mode) (symbol-value mode)))
                minor-mode-list))

  (defun systemhalted/memory-summary ()
    "Return a compact memory summary for diagnostics."
    (cond
     ((fboundp 'memory-use-counts)
      (memory-use-counts))
     (t
      (car (garbage-collect)))))

  (defun systemhalted/diagnostics ()
    "Return runtime diagnostics for investigating slow or memory-heavy sessions."
    (interactive)
    (let ((report
           `((uptime-seconds . ,(float-time (time-subtract (current-time) before-init-time)))
             (current-buffer . ,(buffer-name))
             (major-mode . ,major-mode)
             (buffer-file . ,buffer-file-name)
             (buffer-size . ,(buffer-size))
             (line-count . ,(count-lines (point-min) (point-max)))
             (longest-line-sample . ,(systemhalted/buffer-longest-line-sample))
             (display-line-numbers . ,(bound-and-true-p display-line-numbers-mode))
             (active-minor-modes . ,(systemhalted/active-minor-modes))
             (lsp-mode . ,(bound-and-true-p lsp-mode))
             (timers . ,(length timer-list))
             (idle-timers . ,(length timer-idle-list))
             (garbage-collections . ,gcs-done)
             (memory . ,(systemhalted/memory-summary)))))
      (when (called-interactively-p 'interactive)
        (message "%s" report))
      report))

  (defun systemhalted/emergency-ui-reset ()
    "Disable expensive UI/runtime features and run garbage collection."
    (interactive)
    (systemhalted/display-line-numbers-disable-all)
    (when (bound-and-true-p global-auto-revert-mode)
      (global-auto-revert-mode -1))
    (dolist (buffer (buffer-list))
      (with-current-buffer buffer
        (when (bound-and-true-p flycheck-mode)
          (ignore-errors (flycheck-mode -1)))
        (when (bound-and-true-p lsp-mode)
          (ignore-errors (lsp-disconnect)))
        (when (bound-and-true-p mixed-pitch-mode)
          (ignore-errors (mixed-pitch-mode -1)))
        (when (bound-and-true-p olivetti-mode)
          (ignore-errors (olivetti-mode -1)))))
    (garbage-collect)
    (message "Emergency UI reset complete: %S" (systemhalted/memory-summary)))

  (global-set-key (kbd "C-c !") #'systemhalted/emergency-ui-reset)
#+end_src

** Small convenience keys and windmove

These micro-tweaks live alone because they don't belong with any package:

- =C-x k= rebinds =kill-buffer= → =kill-current-buffer=. The stock binding
  prompts even when point is in the buffer you want to kill, which adds a
  =RET= for nothing.
- =s-u= binds =revert-buffer=, reloading the current buffer from disk. This
  is handy when an external tool rewrites the file underneath you — rustfmt
  on a cargo command, a git rebase, a generated file — since the recursive
  LSP file-watchers are disabled (see the LSP-mode section), so such changes
  don't refresh the buffer on their own. Note the Super key: on macOS
  Command is Super by default (=ns-command-modifier=), so =s-u= is reachable;
  on Linux desktops Super is usually grabbed by the window manager (e.g.
  GNOME Activities), so the chord may not reach Emacs there — fall back to
  =M-x revert-buffer= (or =C-x x g=, =revert-buffer-quick=, which skips the
  confirmation prompt).
- =windmove= is built-in and provides directional window navigation
  (=S-<arrow>= by default). The =use-package= form just enables it.

#+begin_src emacs-lisp
  (global-set-key (kbd "C-x k") #'kill-current-buffer)
  (global-set-key (kbd "s-u") #'revert-buffer)

  ;; enable windmove, for easy navigation between open windows
  (use-package windmove)
#+end_src

** Reload and visit the literate file

This config is iterated on inside Emacs, so the two commands below close
the loop:

- =C-c e= (=systemhalted/config-visit=) opens =systemhalted.org=.
- =C-c r= (=systemhalted/config-reload=) saves the open Org buffer if
  modified, then re-runs =org-babel-load-file= on it. Equivalent to
  restarting Emacs for most config edits, without losing window/buffer
  state. Errors during reload land in =*Messages*= but don't take down
  the session.

#+begin_src emacs-lisp
  (defun systemhalted/config-visit ()
    "Open the literate Emacs configuration."
    (interactive)
    (find-file (expand-file-name "systemhalted.org" user-emacs-directory)))
  (global-set-key (kbd "C-c e") #'systemhalted/config-visit)

  (defun systemhalted/config-reload ()
    "Save and reload the literate Emacs configuration."
    (interactive)
    (let* ((config-file (expand-file-name "systemhalted.org" user-emacs-directory))
           (config-buffer (get-file-buffer config-file)))
      (when (and config-buffer (buffer-modified-p config-buffer))
        (with-current-buffer config-buffer
          (save-buffer)))
      (message "Reloading Emacs configuration...")
      (org-babel-load-file config-file)
      (message "Reloaded Emacs configuration.")))
  (global-set-key (kbd "C-c r") #'systemhalted/config-reload)
#+end_src

** Tutorial dispatcher

=C-h T= mirrors =C-h t= (the built-in Emacs tutorial) but opens subtrees
from /this/ file as read-only indirect buffers. The docs live next to the
code that binds them, the buffer renders as Org so tables, links, and
verbatim keys stay legible, and edits to the source file are immediately
visible because the buffer is indirect — no copy, no drift.

=systemhalted/tutorials= is the registry. Adding a tutorial requires *both*
a new alist entry here *and* a matching =**= heading elsewhere in this file
whose text equals the cdr of the entry; =systemhalted/tutorial= refuses to
open a heading it cannot find. The currently registered tutorials are the
three subtrees in this document: =Tutorial: Organicely — Org-based Second
Brain=, =Using LSP and DAP=, and =Tutorial: Using LaTeX=.

#+begin_src emacs-lisp
  (defvar systemhalted/tutorials
    '(("Organicely"  . "Tutorial: Organicely — Org-based Second Brain")
      ("LSP and DAP" . "Using LSP and DAP")
      ("LaTeX"       . "Tutorial: Using LaTeX"))
    "Alist of (DISPLAY-NAME . ORG-HEADING) tutorial subtrees in systemhalted.org.")

  (defun systemhalted/tutorial ()
    "Pick a tutorial subtree from systemhalted.org and view it read-only.
  Mirrors `help-with-tutorial' (C-h t): a registry of named tutorials is
  offered via `completing-read', and the chosen subtree is shown as a
  narrowed, read-only indirect buffer over the literate config so edits
  to the source remain visible without losing place."
    (interactive)
    (let* ((choice (completing-read "Tutorial: "
                                    (mapcar #'car systemhalted/tutorials)
                                    nil t))
           (heading (cdr (assoc choice systemhalted/tutorials)))
           (buf-name (format "*Tutorial: %s*" choice)))
      (if-let ((existing (get-buffer buf-name)))
          (pop-to-buffer existing)
        (let* ((org-file (expand-file-name "systemhalted.org" user-emacs-directory))
               (base (find-file-noselect org-file))
               (pos (with-current-buffer base
                      (org-find-exact-headline-in-buffer heading))))
          (unless pos
            (user-error "Tutorial heading %S not found in %s" heading org-file))
          (let ((indirect (make-indirect-buffer base buf-name t)))
            (with-current-buffer indirect
              (goto-char pos)
              (org-narrow-to-subtree)
              (org-show-subtree)
              (goto-char (point-min))
              (read-only-mode 1))
            (pop-to-buffer indirect))))))
  (global-set-key (kbd "C-h T") #'systemhalted/tutorial)
#+end_src

** Dired, fonts, and theme

Three small finishing touches:

- *Dired.* =c= in a Dired buffer creates an empty file (overriding the
  stock binding for =dired-do-copy=, which uses =C= anyway). One-key file
  creation is the move most often made from a Dired listing.
- *Fonts.* Fira Code for fixed-pitch (default face) and Cantarell for
  variable-pitch when both are installed. =mixed-pitch-mode= consumes the
  variable-pitch face for the book-style reading view further down. Sizes
  are =150= (=15pt=) on both faces; the =find-font= guards make the block
  no-op when a font isn't installed instead of leaving a missing-font
  warning.
- *Theme.* Nord. =systemhalted/load-theme= disables every currently-enabled
  theme first because repeated config reloads otherwise stack faces or
  leave stale theme state behind. =nord-region-highlight 'frost= is the
  one face customization — selection in the bluer "frost" palette rather
  than the default muted gray.

#+begin_src emacs-lisp
  (with-eval-after-load 'dired
    (define-key dired-mode-map (kbd "c") #'dired-create-empty-file))

  (when (display-graphic-p)
    (when (find-font (font-spec :name "Fira Code"))
      (set-face-attribute 'default nil :font "Fira Code" :height 150))
    (when (find-font (font-spec :name "Cantarell"))
      (set-face-attribute 'variable-pitch nil :font "Cantarell" :height 150)))

  (defun systemhalted/load-theme ()
    "Load the preferred theme without stacking themes on reload."
    (mapc #'disable-theme custom-enabled-themes)
    (load-theme 'nord t))

  (use-package nord-theme
    :ensure t
    :custom
    (nord-region-highlight "frost")
    :config
    (systemhalted/load-theme))
#+end_src

** Clipboard

GUI Emacs already shares the system clipboard with the host on every windowed
backend it supports (NS/Cocoa, X11, PGTK, Wayland). The options below add the
polish that turns that bidirectional plumbing into the behavior most people
expect:

- =save-interprogram-paste-before-kill= pushes whatever currently sits in the
  host clipboard onto the kill-ring /before/ a new Emacs kill overwrites the
  selection, so a paste from outside Emacs is still recoverable through =M-y= /
  =consult-yank-pop=.
- =yank-pop-change-selection= mirrors the result of =M-y= back to the system
  clipboard so the next external paste matches what is visible in Emacs.
- =kill-do-not-save-duplicates= collapses adjacent identical kills in the
  kill-ring so =M-y= history is not cluttered by repeated yanks of the same
  thing.

Terminal Emacs (=emacs -nw= inside Terminal.app, iTerm2, GNOME Terminal,
Alacritty, etc.) talks to no display server and therefore cannot reach the
host clipboard without help. =xclip= bridges the gap: on macOS it shells out
to =pbcopy= / =pbpaste=, on X11 to =xclip= or =xsel=, on Wayland to =wl-copy=
/ =wl-paste=. It is loaded only when the starting frame is non-graphical so
it does not displace the native bridge in windowed GUI Emacs.

#+begin_src emacs-lisp
(setq select-enable-clipboard t
      select-enable-primary nil
      save-interprogram-paste-before-kill t
      yank-pop-change-selection t
      kill-do-not-save-duplicates t)

(use-package xclip
  :ensure t
  :if (not (display-graphic-p))
  :config
  (xclip-mode 1))
#+end_src

** Text Manipulation

Three keys for the line-shape edits that come up in every coding session:
reorder, duplicate, and join. Each is small enough to belong at the editor
layer rather than inside a structural-editing package like Paredit or
Combobulate.

*** Move text up or down

=move-text= bubbles the current line (or active region) up with =M-p= or
down with =M-n=, preserving indentation. Easier than the cut-and-yank
two-step and faster than =transpose-lines=, which only works on adjacent
single lines and leaves point in an awkward place. The =message= in
=:config= confirms the package loaded — useful when iterating on this
file with =C-c r=.

#+begin_src emacs-lisp
 (use-package move-text
  :ensure t
  :bind (("M-n" . move-text-down)
         ("M-p" . move-text-up))
  :config
  (message "move-text loaded successfully"))
#+end_src

*** Duplicate the current line or region

=C-c d= copies the current line (or active region) below itself, leaving
point at the same column on the new copy. With a numeric prefix it makes
=N= copies; with a negative prefix it duplicates and comments out the
original — handy for keeping a known-good version while editing a copy.
The IntelliJ =Cmd-D= reflex, ported. Source:
https://www.emacswiki.org/emacs/CopyingWholeLines#toc12

#+begin_src emacs-lisp
     (define-key global-map (kbd "C-c d")
  (defun systemhalted/duplicate-line-or-region (&optional n)
    "Duplicate current line, or region if active.
    With argument N, make N copies.
    With negative N, comment out original line and use the absolute value."
    (interactive "*p")
    (let ((use-region (use-region-p)))
      (save-excursion
        (let ((text (if use-region        ;Get region if active, otherwise line
                        (buffer-substring (region-beginning) (region-end))
                      (prog1 (thing-at-point 'line)
                        (end-of-line)
                        (if (< 0 (forward-line 1)) ;Go to beginning of next line, or make a new one
                            (newline))))))
          (dotimes (i (abs (or n 1)))     ;Insert N times, or once if not specified
            (insert text))))
      (if use-region nil                  ;Only if we're working with a line (not a region)
        (let ((pos (- (point) (line-beginning-position)))) ;Save column
          (if (> 0 n)                             ;Comment out original with negative arg
              (comment-region (line-beginning-position) (line-end-position)))
          (forward-line 1)
          (forward-char pos))))))

#+end_src


*** Join following line or unwrap region

=C-c k= folds the next line up into the current one, replacing the line
break with a single space. With a region active it unwraps the whole
region into a single line by calling =fill-region= with an unreachable
=fill-column=. The two cases share a binding because they are the same
intent — "make this one line" — at different scales.

#+begin_src emacs-lisp
  (define-key global-map (kbd "C-c k")
  (defun systemhalted/join-following-line (arg)
    "Joins the following line or the whole selected region"
    (interactive "P")
    (if (use-region-p)
        (let ((fill-column (point-max)))
          (fill-region (region-beginning) (region-end)))
      (join-line -1))))
#+end_src


* Completion, Navigation, and Help

The completion stack is built from small packages that cooperate through
Emacs' standard completion system and adjacent help/navigation APIs rather than
a single large framework. Each piece has one job:

- =Savehist= persists minibuffer, search, kill-ring, and completion history.
- =Orderless= decides /what counts as a match/ for a typed query.
- =Vertico= renders /minibuffer/ candidates (file pickers, =M-x=, prompts).
- =Marginalia= annotates those minibuffer candidates with extra context.
- =Consult= supplies jump-and-search commands that feed the minibuffer.
- =Embark= acts on the current candidate or symbol without leaving context.
- =Helpful= renders richer reference buffers for commands, variables, keys, and symbols.
- =Corfu= renders /in-buffer/ completion (the popup while typing code).
- =Projectile= adds project-scoped commands (find file in project, etc.).
- =ibuffer= manages buffers in bulk (mark, filter, kill) at =C-x C-b=.
- =which-key= pops up keymap continuations after any prefix key.

Two surfaces, two UIs: minibuffer completion (Vertico) is intentionally
separate from at-point completion (Corfu). Both reuse the same matching style
through Orderless. Help and discovery sit beside those completion surfaces:
Embark exposes available actions, Helpful explains Lisp objects, and which-key
shows prefix maps. Any one piece can be swapped or removed without disturbing
the others.

** Savehist — persist minibuffer history

[[https://www.gnu.org/software/emacs/manual/html_node/emacs/Minibuffer-History.html][Savehist]] persists minibuffer history across restarts. It is the substrate
the rest of this section sits on: =vertico= sorts candidates by recency,
=consult-yank-pop= reads =kill-ring=, and =corfu-history= persists itself
through =savehist-additional-variables=. Bumping =history-length= to 200
keeps enough recent entries to be useful without bloating =history=.

#+begin_src emacs-lisp
(use-package savehist
  :ensure nil
  :custom
  (savehist-additional-variables '(search-ring regexp-search-ring kill-ring))
  (history-length 200)
  :init
  (savehist-mode 1))
#+end_src

** Orderless — flexible matching style

[[https://github.com/oantolin/orderless][Orderless]] is a completion /style/, not a UI. It changes how the typed
input is matched against candidates: whitespace-separated components match
in any order, anywhere in the candidate. Typing =kafka proj= can match
=src/projects/kafka-bridge.clj= even though those tokens are not adjacent
and not in that order. Every minibuffer command (Vertico) and every at-point
popup (Corfu) inherits this behavior because they both ask Emacs' generic
completion machinery to filter, and that machinery consults
=completion-styles=.

The =basic= fallback is kept for cases like TRAMP paths where strict prefix
matching is required, and =partial-completion= is added for file names so
=/u/s/l= still expands to =/usr/share/local=.

=orderless-matching-styles= adds =orderless-prefixes= to the default chain.
That style splits a single token on =-= or =_= and matches each piece as a
prefix anywhere in the candidate, so =M-x desc-fun= finds
=describe-function=, =sql-co-bu= finds =sql-connect-buffer=, etc. — the
behavior most people expect when typing a hyphenated symbol from memory.
=orderless-literal= still wins when the token contains characters that
would not parse as a regexp, and =orderless-regexp= remains as the final
fallback.

#+begin_src emacs-lisp
(use-package orderless
  :ensure t
  :custom
  (completion-styles '(orderless basic))
  (orderless-matching-styles '(orderless-literal
                               orderless-prefixes
                               orderless-regexp))
  (completion-category-defaults nil)
  (completion-category-overrides '((file (styles basic partial-completion)))))
#+end_src

** Built-in completion defaults

A few built-in switches make the small-package stack feel polished without
adding any new package. =completion-ignore-case= and its file/buffer
counterparts let Orderless tokens match regardless of case: =M-x desc-fun=
finds =describe-function=. =enable-recursive-minibuffers= allows nested
prompts (=M-:= from inside =find-file=, or any Embark action that opens a
new minibuffer); =minibuffer-depth-indicate-mode= pairs with it so the
prompt shows a =[2]=, =[3]= prefix when nested — without that, =C-g= can
look like it "exits completely" when really it just dropped one level.
The =minibuffer-prompt-properties= addition plus =cursor-intangible-mode=
prevents the cursor from accidentally landing inside the prompt text.

#+begin_src emacs-lisp
(use-package emacs
  :ensure nil
  :custom
  (completion-ignore-case t)
  (read-buffer-completion-ignore-case t)
  (read-file-name-completion-ignore-case t)
  (enable-recursive-minibuffers t)
  (minibuffer-prompt-properties
   '(read-only t cursor-intangible-mode t face minibuffer-prompt))
  :hook (minibuffer-setup . cursor-intangible-mode)
  :init
  (minibuffer-depth-indicate-mode 1))
#+end_src

** Vertico — minibuffer UI

[[https://github.com/minad/vertico][Vertico]] is what shows up whenever Emacs reads something interactively:
=C-x C-f= (find file), =C-x b= (switch buffer), =M-x= (run command),
=C-h f= (describe function), capture template selection, yes/no prompts,
and so on. Candidates appear as a vertical list under the prompt; =C-n= /
=C-p= move the selection, =RET= accepts, =TAB= completes the common prefix.
=vertico-cycle t= lets the selection wrap around at the ends of the list.

Vertico does not decide /which/ candidates match — Orderless does that.
Vertico just displays the filtered set and tracks the current choice.

#+begin_src emacs-lisp
(use-package vertico
  :ensure t
  :custom
  (vertico-cycle t)
  :config
  (vertico-mode 1))
#+end_src

** Vertico-Directory — Ivy-style file path navigation

[[https://github.com/minad/vertico/blob/main/extensions/vertico-directory.el][Vertico-Directory]] is an extension that ships with Vertico. It rebinds three
keys inside =vertico-map= so the minibuffer behaves like a file browser when
the prompt holds a path:

- =RET= on a directory candidate /descends into/ that directory and lists
  its contents, instead of selecting it as a literal value. Pair with =M-RET=
  (Vertico default) when you really do want the literal path.
- =C-l= moves /up/ one directory regardless of where point sits in the
  prompt — the Ivy =counsel-find-file= reflex.
- =DEL= at the end of =~/.emacs.d/foo/= deletes the trailing path component
  back to =~/.emacs.d/=, the way Ivy's =counsel-find-file= behaved. Outside
  that context it is an ordinary backspace.
- =M-DEL= deletes a word inside the path.

The =rfn-eshadow-update-overlay= hook installs =vertico-directory-tidy= so
shadowed prefixes collapse cleanly: typing =/etc= after =~/foo/= shows
=/etc=, not =~/foo//etc=. The extension lives inside the installed Vertico
package, so =:ensure nil :after vertico= is correct — no separate install.

#+begin_src emacs-lisp
(use-package vertico-directory
  :ensure nil
  :after vertico
  :bind (:map vertico-map
              ("RET"   . vertico-directory-enter)
              ("C-l"   . vertico-directory-up)
              ("DEL"   . vertico-directory-delete-char)
              ("M-DEL" . vertico-directory-delete-word))
  :hook (rfn-eshadow-update-overlay . vertico-directory-tidy))
#+end_src

** Hidden file toggle in file prompts

Pressing =M-h= inside any file prompt flips visibility of dotfiles for the
current and subsequent prompts. The implementation is small:
=systemhalted-vertico-show-hidden-files= is a global flag (defaults to =t=
to match plain =read-file-name= behavior — show everything). An =:around=
advice on =read-file-name= installs a wrapping predicate that drops dotfiles
when the flag is nil, /except/ when the basename being typed in the prompt
already starts with =.= — that escape hatch lets you reach a hidden file
without first hitting the toggle.

The toggle command flips the flag and forces Vertico to re-filter the
current candidate set in place by invalidating its input cache and calling
=vertico--exhibit=, so you don't have to abort and re-prompt to see the
effect. =M-h= is bound in =vertico-map=, not in
=minibuffer-local-filename-completion-map=, because Vertico's setup hook
replaces the local map outright (=use-local-map vertico-map=); the latter
map is never consulted during a Vertico-driven file prompt. The command
guards itself with =minibuffer-completing-file-name= so pressing =M-h= in a
buffer / command / variable prompt is a clear no-op rather than a surprise.

#+begin_src emacs-lisp
(defvar systemhalted-vertico-show-hidden-files t
  "When non-nil, file-name completion shows hidden files (dotfiles).
Toggle with `systemhalted/vertico-toggle-hidden-files' inside a file prompt.")

(defun systemhalted--file-name-hidden-p (file)
  "Return non-nil if FILE's basename is a dotfile (excluding `.' and `..')."
  (let ((name (file-name-nondirectory (directory-file-name file))))
    (and (string-prefix-p "." name)
         (not (member name '("." ".."))))))

(defun systemhalted--read-file-name-with-hidden-filter
    (orig prompt &optional dir default-filename mustmatch initial predicate)
  "Around-advice for `read-file-name' that hides dotfiles per the toggle."
  (let ((merged
         (lambda (f)
           (and (or systemhalted-vertico-show-hidden-files
                    (string-prefix-p
                     "."
                     (file-name-nondirectory
                      (or (and (minibufferp)
                               (minibuffer-contents-no-properties))
                          "")))
                    (not (systemhalted--file-name-hidden-p f)))
                (or (null predicate) (funcall predicate f))))))
    (funcall orig prompt dir default-filename mustmatch initial merged)))

(advice-add 'read-file-name :around
            #'systemhalted--read-file-name-with-hidden-filter)

(defun systemhalted/vertico-toggle-hidden-files ()
  "Toggle visibility of hidden files in the current minibuffer file prompt.
Flips `systemhalted-vertico-show-hidden-files' and refreshes Vertico in
place when called from inside an active file prompt. No-op on non-file
prompts so the keybinding does not surprise in `M-x' or buffer prompts."
  (interactive)
  (cond
   ((and (minibufferp) (not minibuffer-completing-file-name))
    (message "Hidden-file toggle only applies to file prompts"))
   (t
    (setq systemhalted-vertico-show-hidden-files
          (not systemhalted-vertico-show-hidden-files))
    (when (and (minibufferp) (boundp 'vertico--input))
      (setq vertico--input t)
      (when (fboundp 'vertico--exhibit)
        (vertico--exhibit)))
    (message "Hidden files: %s"
             (if systemhalted-vertico-show-hidden-files "shown" "hidden")))))

(with-eval-after-load 'vertico
  (define-key vertico-map (kbd "M-h")
              #'systemhalted/vertico-toggle-hidden-files))
#+end_src

** Marginalia — annotations in the minibuffer

[[https://github.com/minad/marginalia][Marginalia]] adds a right-aligned annotation column to Vertico's candidates.
The annotation depends on the candidate type: commands show their docstring
summary and key binding, files show mode/size/mtime, buffers show their
major mode and file path, variables show their current value. This turns
=M-x= and =describe-variable= into a passive reference — you can browse
without committing to a selection.

It loads after Vertico because the annotation hook is only useful once a
minibuffer UI is active. The default =marginalia-annotators= registry
already lists the rich annotator first for each candidate category, so
full annotations show on the first prompt — no override required. =M-A=
inside the minibuffer cycles each category to its lighter variants
(=builtin=, =none=) when the default annotation feels too long.

#+begin_src emacs-lisp
(use-package marginalia
  :ensure t
  :after vertico
  :bind (:map minibuffer-local-map
              ("M-A" . marginalia-cycle))
  :init
  (marginalia-mode 1))
#+end_src

** Consult — focused search and navigation commands

[[https://github.com/minad/consult][Consult]] is a collection of commands purpose-built to feed the
Vertico-style minibuffer. They take something Emacs already knows about
(buffer lines, open buffers, project files, recent files, bookmarks, the
kill ring, marks, …) and expose it as a narrowable, previewable list.

The daily bindings here:

- =C-s= runs =consult-line= — incremental search across the current
  buffer's lines, but each candidate is a full line so Orderless tokens
  apply. Better than =isearch= when the goal is /jump to/ rather than
  /step through/.
- =C-x b= and =C-c b= run =consult-buffer= — a unified picker over open
  buffers, recent files, and bookmarks. Type the narrowing key (=b=, =f=,
  =m=, =p=, …) at the start to restrict the source.
- =C-c s= runs =consult-ripgrep= — streams =rg= results into the
  minibuffer with live preview. The de facto project search.
- =M-y= runs =consult-yank-pop= — pick from the kill ring as a previewable
  list instead of cycling blindly. Persisted across restarts because
  =savehist-additional-variables= keeps =kill-ring=.
- =M-g g= and =M-g M-g= remap to =consult-goto-line=, which previews the
  destination line as you type the number.

Both =C-x b= and =C-c b= bind the same command on purpose: =C-x b= matches
Emacs muscle memory, =C-c b= sits in the user prefix range alongside the
other =C-c= shortcuts.

=consult-narrow-key= is set to =<= so a leading =<= followed by a narrowing
character restricts the source (e.g. =< b= for buffers in =consult-buffer=).
The =xref-show-*-function= hooks route =M-.= and =xref-find-references=
through Consult, so =Vertico= replaces the legacy =*xref*= buffer when
multiple candidates exist. Previews are debounced via =consult-customize=:
buffers/recent files preview after 0.2s of stillness, ripgrep/grep/line
previews wait 0.4s, which keeps large repos from churning the display while
you scroll candidates.

#+begin_src emacs-lisp
(use-package consult
  :ensure t
  :bind
  (("C-s"   . consult-line)
   ("C-x b" . consult-buffer)
   ("C-c b" . consult-buffer)
   ("C-c s" . consult-ripgrep)
   ("M-y"   . consult-yank-pop)
   ([remap goto-line] . consult-goto-line))
  :custom
  (consult-narrow-key "<")
  (xref-show-xrefs-function       #'consult-xref)
  (xref-show-definitions-function #'consult-xref)
  :config
  (consult-customize
   consult-buffer consult-recent-file
   :preview-key '(:debounce 0.2 any)
   consult-ripgrep consult-grep consult-line
   :preview-key '(:debounce 0.4 any)))
#+end_src

** Embark — act on minibuffer candidates

[[https://github.com/oantolin/embark][Embark]] adds the verb that the rest of this stack was missing. With a
candidate selected in any minibuffer prompt, =C-.= opens a context-sensitive
action menu: open in another window, copy path, kill the buffer, browse the
URL, run grep on the directory, describe the symbol, etc. =C-;= is
=embark-dwim=, which runs the most likely action without showing the menu.
=C-h B= lists every action bound for the current candidate type, which is
how to discover what's available without leaving the prompt.

Setting =prefix-help-command= to =embark-prefix-help-command= replaces the
default =C-h-after-prefix= help with a Vertico-rendered prompt: pressing
=C-h= after =C-x= shows every =C-x= binding as a searchable list.

The integration with Consult is built into Embark: =embark-export= from a
=consult-line= or =consult-ripgrep= result produces an editable grep-mode
buffer, and =consult-preview-at-point-mode= previews the underlying
location as you move through an =embark-collect= buffer.

#+begin_src emacs-lisp
(use-package embark
  :ensure t
  :bind
  (("C-."   . embark-act)
   ("C-;"   . embark-dwim)
   ("C-h B" . embark-bindings))
  :custom
  (prefix-help-command #'embark-prefix-help-command)
  :config
  ;; Consult integration is built into embark; just enable the preview hook.
  (with-eval-after-load 'consult
    (add-hook 'embark-collect-mode-hook #'consult-preview-at-point-mode)))
#+end_src

** Helpful — richer describe buffers

[[https://github.com/Wilfred/helpful][Helpful]] replaces the built-in
=describe-*= buffers with richer Emacs Lisp reference pages. It does not
participate in completion or navigation directly; Vertico, Consult, Embark,
Corfu, Projectile, and Xref still own those jobs. Helpful is for the moment
after a command, variable, key, or symbol has been found and you want better
documentation: source links, aliases, callers, key bindings, plist details,
and a more useful summary of what kind of Lisp object is being described.

The standard help keys are remapped rather than duplicated, so =C-h f=,
=C-h v=, and =C-h k= keep their muscle memory while opening Helpful buffers.
=C-h f= goes to =helpful-callable= instead of =helpful-function= because a
symbol at that prompt may be a function, macro, or special form. Direct
bindings for =helpful-function= and =helpful-symbol= remain on =C-h F= and
=C-h S= for cases where the narrower command is wanted.

=C-h .= is intentionally reassigned from =display-local-help= to
=helpful-at-point=. This configuration is maintained mostly by editing
Emacs Lisp in =systemhalted.org=, so "describe the symbol under point" is
more useful than showing button or widget =help-echo= text. The built-in
local help command remains available on =C-h C-.= for the occasional UI
button/widget case.

Programming-language hover documentation stays under LSP: use =C-c l h h=
for Java, Python, TypeScript, and other language-server buffers. Helpful is
for Emacs Lisp symbols, commands, variables, faces, and keys.

#+begin_src emacs-lisp
(use-package helpful
  :ensure t
  :bind
  (([remap describe-function] . helpful-callable)
   ([remap describe-command]  . helpful-command)
   ([remap describe-variable] . helpful-variable)
   ([remap describe-key]      . helpful-key)
   ("C-h ."                   . helpful-at-point)
   ("C-h C-."                 . display-local-help)
   ("C-h F"                   . helpful-function)
   ("C-h S"                   . helpful-symbol)))
#+end_src

** Corfu — in-buffer completion popup

[[https://github.com/minad/corfu][Corfu]] is the popup that appears /inside/ a buffer while typing code:
function names, local variables, LSP suggestions, snippet keys, dabbrev
matches. It is the in-buffer counterpart to Vertico, and it consumes
candidates from =completion-at-point-functions= (capf) — the same hook
that =lsp-mode=, =yasnippet=, and the Emacs builtins write to. Whatever
the active mode contributes to capf, Corfu shows.

With =corfu-auto t= the popup appears automatically after =corfu-auto-delay=
(=0.35s=) once the typed prefix is at least =corfu-auto-prefix= characters
(=3=). The slightly slower trigger is intentional: Java LSP completion can
allocate a lot of Emacs-side JSON and candidate metadata, so completion waits
until the prefix is specific enough to be useful. =TAB= accepts the selection,
=M-n= / =M-p= cycle, =corfu-cycle t= wraps at the ends, and
=corfu-preselect 'prompt= keeps your typed prefix selected so =RET= inserts
what you typed rather than the first suggestion. =global-corfu-mode= turns it
on in every buffer.

Corfu replaces =company= from the older config; see the Programming section
for how =lsp-mode= is configured to feed Corfu through capf.

#+begin_src emacs-lisp
(use-package corfu
  :ensure t
  :custom
  (corfu-auto t)
  (corfu-auto-delay 0.35)
  (corfu-auto-prefix 3)
  (corfu-cycle t)
  (corfu-preselect 'prompt)
  :config
  (global-corfu-mode 1))
#+end_src

** Projectile — project-scoped commands

[[https://github.com/bbatsov/projectile][Projectile]] knows what a /project/ is — any directory containing a =.git=,
a recognized build file, or a =.projectile= marker — and exposes commands
that operate over that scope as a unit: switch project, find file in
project, kill project buffers, run the project's tests, search/replace
across the project, etc. The full keymap lives under the conventional
=C-c p= prefix, which is why other custom bindings in this config avoid
lowercase =C-c p= entirely and take capitals (=C-c B=) or other letters
(=C-c n=) instead.

=projectile-completion-system 'default= keeps Projectile's prompts going
through Emacs' standard completion — meaning Vertico + Orderless +
Marginalia all apply to project pickers too. =projectile-indexing-method
'alien= delegates the file listing to external tools (=git ls-files=,
=find=) instead of Emacs traversing directories itself; on large
repositories this is the difference between instant and seconds-long
listings.

Projects here don't sit in one flat directory — they're nested a few
levels deep under =~/Workspaces= (e.g. =~/Workspaces/Palak/products/…=).
Projectile's built-in =projectile-project-search-path= only scans a
single, fixed depth, so it can't find them without hardcoding how deep
each one lives. Instead =systemhalted/discover-projects= walks
=systemhalted/projects-root= (=~/Workspaces=) recursively to /any/ depth
and registers every repo it finds — stopping the moment it hits a project
root so it never descends into a repo's own file tree. It runs on startup,
so =C-c p p= lists every project without first visiting a file inside each
one, and it's guarded on the directory existing so it's a harmless no-op on
machines without that tree. After adding a new repo, re-scan on the spot
with =M-x systemhalted/discover-projects= — no restart needed.

Common entry points (all under =C-c p=):

- =C-c p p= — switch project.
- =C-c p f= — find file in current project.

** compile — multiple compilation buffers

Projectile's run commands (=C-c p u=) use Emacs' built-in =compile=
infrastructure. By default, all compilation commands share a single
=*compilation*= buffer, which means running a long-lived process like
=mvn spring-boot:run= blocks any subsequent =mvn test= invocation.

Setting =compilation-buffer-name-function= to generate unique buffer names
allows multiple compilation processes to run in parallel. Each buffer is
named after the command that spawned it, making it easy to identify which
process is which when switching buffers.

#+begin_src emacs-lisp
(use-package compile
  :ensure nil
  :custom
  (compilation-scroll-output t)
  :config
  (defun systemhalted/compilation-buffer-name (mode)
    "Generate a unique compilation buffer name based on the command.
MODE is the major mode name passed by `compilation-start'."
    (let* ((cmd (or compilation-arguments "compile"))
           ;; Truncate long commands for readability
           (short-cmd (if (> (length cmd) 40)
                          (concat (substring cmd 0 37) "...")
                        cmd)))
      (generate-new-buffer-name (format "*%s: %s*" mode short-cmd))))
  (setq compilation-buffer-name-function #'systemhalted/compilation-buffer-name))
#+end_src

** which-key — keymap discovery

[[https://github.com/justbur/emacs-which-key][which-key]] is a passive
keymap aid. Press a prefix key — =C-c=, =C-x=, =C-h=, =C-x 4=, =C-c p= for
Projectile — and after a short idle delay a popup lists every continuation
bound under that prefix together with the command it runs. This makes the
rest of this section's bindings self-documenting: typing =C-c= and waiting
reveals =C-c b=, =C-c s=, =C-c p=, =C-c r=, =C-c c=, =C-c n=, etc., without
having to remember or look them up.

side-window cap to half the frame gives them room to render without
wrapping. The popup still shrinks to fit when fewer keys are bound.

* Org Mode

Org here is /Organicely/, an Org-based second brain — notes only, no task
management. The single
sorting rule, borrowed from [[https://systemhalted.in/2026/05/08/saving-private-notes/][Saving Private Notes]]:

#+begin_example
If it has a "done" state, it goes in Reminders.
If it has a date, it goes in Calendar.
Everything else goes in org.
#+end_example

That rule replaced the previous three-file TODO/BACKLOG/NOTES model (which
itself replaced an older GTD-heavy setup). Tasks and scheduling now live
entirely in Apple Reminders and Calendar; org models /knowledge/, organized as
a directory tree under =~/organicely/= — one file per note, in numbered areas
that mirror the Apple Notes folders:

#+begin_example
00-inbox/      -> unfiled captures, emptied weekly
10-work/       -> work notes
20-personal/   -> career/ home/ health/ family/ finances/ purchases/ recommendations/
30-learning/   -> ai-sessions/ articles/ books/ podcasts/ courses/ videos/ subjects/
40-writing/    -> concepts/ ideas/ drafts/ articles/ podcasts/ poetry/ stories/ books/ published/
50-journal/    -> daily/ (one file per day)   old/ (migrated entries)
60-ideas/      -> apps-and-tools/ business/ research-questions/
99-archive/    -> done thinking, kept searchable
#+end_example

A note is a file. Capture (=C-c c=) creates one directly in an area/section, or
in the inbox when filing would interrupt the thought. Apple Notes keeps exactly
one folder — its Inbox — as the mobile capture point; a weekly half-hour sweep
moves those items here and refiles =00-inbox/= with =C-c C-w= (which relocates
the note's file into place). There is no agenda, no TODO lifecycle, and no
promotion command — only a soft save-time warning when a task-shaped heading
sneaks into =~/organicely/=.

Jupyter support is opt-in. Normal Org startup should stay light and quiet; when a
notebook workflow is needed, =C-c j= enables the Jupyter Babel integration for
that session.

Org export is available for common document formats:

- =C-c C-e m m= exports the current Org file to Markdown.
- =C-c C-e h h= exports the current Org file to HTML.
- =C-c C-e l l= exports the current Org file to LaTeX source.
- =C-c C-e l p= exports the current Org file to PDF through LaTeX.

For this configuration, run those commands from =systemhalted.org=. Markdown,
HTML, and LaTeX source exports are written next to the Org file as
=systemhalted.md=, =systemhalted.html=, and =systemhalted.tex=. PDF export uses
the =latexmk= setup in the LaTeX section.

** Tutorial: Organicely — Org-based Second Brain

One rule decides where everything goes:

#+begin_example
Has a "done" state -> Apple Reminders
Has a date         -> Apple Calendar
Everything else    -> org
#+end_example

Org never holds tasks. "Buy milk" is a Reminder; "Why CRDTs converge" is a
note. When in doubt, ask whether the item can ever be /finished/ — if yes, it
does not belong here.

*The structure.* =~/organicely/= is a directory tree, and every note is its own
file. The numbered areas mirror the folders from Apple Notes; each area holds
section subdirectories, and a note is a single =.org= file with a =#+TITLE:= and
prose under it:

#+begin_example
00-inbox/      unfiled captures (emptied during the weekly sweep)
10-work/       work notes
20-personal/   career/ home/ health/ family/ finances/ purchases/ recommendations/
30-learning/   ai-sessions/ articles/ books/ podcasts/ courses/ videos/ subjects/
40-writing/    concepts/ ideas/ drafts/ articles/ podcasts/ poetry/ stories/ books/ published/
50-journal/    daily/ (one file per day)   old/ (migrated entries)
60-ideas/      apps-and-tools/ business/ research-questions/
99-archive/    done thinking, kept searchable
#+end_example

Filenames are kebab-case slugs of the title (=creatine.org=, =idp-adapter.org=).
Sections are just directories, so a new one appears the moment you file into it.
The area and section directories are created at startup; nothing else is seeded.

*Capture.* =C-c c= (choose =c= Note) asks one question — where does this go? —
then a title:

#+begin_example
Capture to:  Inbox              -> 00-inbox/<timestamp>.org  (title optional; file it later)
             20-personal/health -> 20-personal/health/<title>.org
             10-work/meetings   (type a new area/section to create it on the fly)
#+end_example

Pick =Inbox= when deciding would interrupt the thought; pick an area/section to
file it straight away. Either way a fresh file is created with a
=#+TITLE:=/=#+DATE:= header and left open to write; =C-c C-c= finishes, =C-c C-k=
aborts.

*Journaling.* Journaling is a writing session, not a quick capture, so it keeps
its own key. =C-c J= opens today's entry — =50-journal/daily/YYYY-MM-DD.org= —
creating it (titled e.g. =5th June 2026, Friday=) if needed, alone in the frame
with the book view on. Write prose; =C-x C-s= saves; =C-c B= toggles the book
view back off when done.

*Source notes.* =C-c c s= is source-aware: launch it while point is on a heading
in another Org file and the note lands in =00-inbox/sources.org= under a heading
dedicated to that source. The source heading gains an =:ID:= the first time, the
inbox heading a matching =:SOURCE_ID:=, and every future note from the same
source joins the same subtree — even if the source is later renamed. From a
non-Org buffer it falls back under the flat =* Inbox= heading. The body carries a
backlink (=%a=) so =C-c C-o= jumps back to where you were reading.

*The weekly sweep.* Half an hour, once or twice a week:

1. Move anything in the Apple Notes Inbox into org (capture or paste).
2. Open each file in =00-inbox/= and refile it with =C-c C-w= — this /moves the
   file/ (and its attachments) into a chosen =area/section= directory, completed
   through vertico. Filing into =30-learning/articles= is a few keystrokes; type
   a directory that does not exist yet to create it.
3. Notes about finished thinking move to =99-archive/=.

The inbox should be empty when the sweep ends. The system stays trustworthy
because of this ritual, not because of any enforcement.

*Retrieval.* =C-c n= runs a ripgrep search across =~/organicely/= only (=C-c s=
is the general-purpose search that prompts for a directory) — with one file per
note it drops you straight into the note. Inside a note, =C-c l= stores a link
and =C-c C-l= inserts it, so durable cross-references between notes are cheap.

*The soft warning.* Saving any file under =~/organicely/= that contains a =TODO= /
=IN-PROGRESS= / =DONE= heading or a =SCHEDULED:= / =DEADLINE:= line prints a
reminder that the item belongs in Reminders. The save always succeeds — the
warning is a nudge, not a guard.

*Attachments.* A note's images and files live beside it in a
=<note>-attachments/= directory and are linked from the note; =C-c C-w= carries
them along when you refile.

For code and notebooks, Org Babel loads Emacs Lisp, Java, Python, and shell by
default. Jupyter support is available on demand:

#+begin_example
C-c j
#+end_example

Use Jupyter only when an Org file needs notebook-style execution. Keeping it
opt-in prevents normal Org startup from carrying notebook server state.

** Path helpers and directory vocabulary

These tiny pieces are the vocabulary the rest of the Org setup speaks:
=systemhalted/org-file= resolves a relative name against =org-directory=
(centralises the =~/organicely/= assumption so nothing else hard-codes it — it
takes =area/section= subpaths too), and =systemhalted/org-buffer-file-p= asks
"is the current buffer this specific file?" using =file-truename= so symlinks
into =~/organicely/= behave like the real thing.

=systemhalted/org-area-dirs= names the eight numbered area directories — the
sort-by-prefix structure carried over from the Apple Notes system. The alist
=systemhalted/org-area-sections= records the section subdirectories seeded under
each area, mirroring the sub-folders each area had in Apple Notes; areas not
listed (=10-work=, =99-archive=) start flat. Sections are just directories, so
new ones are created on demand when you capture or refile into them.

=systemhalted/org-done-state-regexp= is the one definition of "task-shaped":
a heading carrying =TODO= / =IN-PROGRESS= / =DONE=, or a =SCHEDULED:= /
=DEADLINE:= planning line. Items matching it have a done state and therefore
belong in Reminders, not here.

=systemhalted/org-slug= turns a title into a kebab-case filename (lowercase,
hyphen-separated, Unicode letters kept) and =systemhalted/org-unique-file= adds
a numeric suffix when that name is taken, so two notes never collide.
=systemhalted/org-section-dirs= lists every =area= and =area/section= directory
(skipping the =-attachments= folders) — the completion candidates that both
capture and refile offer.

=systemhalted/org-capture-file= is the =C-c c= capture target: it asks for a
destination (=Inbox=, or any =area/section= — a new one is created if typed),
then a title (optional for the inbox, where the file is timestamped), creates
the =.org= file with a =#+TITLE:=/=#+DATE:= header, and leaves point ready for
the body. =systemhalted/refile-note= is its weekly-sweep counterpart: it moves
the current note's file — and its =-attachments= folder — into a chosen
=area/section=, the directory-model stand-in for =org-refile= (bound to =C-c C-w=
in Org buffers).

=systemhalted/org-goto-or-create-heading= moves point /onto/ a named top-level
heading in the current buffer, creating it (with surrounding blank lines) when
missing. It survives for the source-aware capture below, which groups every note
about one source under a single heading in =00-inbox/sources.org=.

=systemhalted/journal= (bound to =C-c J=) is the way into journaling — a
sit-down writing session, not a capture. Each day is its own file,
=50-journal/daily/YYYY-MM-DD.org=; the command opens today's (creating it with a
human-readable title like =5th June 2026, Friday= when needed) alone in the
frame with the book view on (olivetti margins + mixed-pitch — defined at the end
of this section) for distraction-free writing. =C-c B= toggles the view back
off; saving is the usual =C-x C-s=.

#+begin_src emacs-lisp
(defun systemhalted/org-file (file)
  "Return FILE expanded inside `org-directory'."
  (expand-file-name file org-directory))

(defun systemhalted/org-buffer-file-p (file)
  "Return non-nil when the current buffer visits FILE in `org-directory'."
  (and buffer-file-name
       (string= (file-truename buffer-file-name)
                (file-truename (systemhalted/org-file file)))))

(defconst systemhalted/org-area-dirs
  '("00-inbox" "10-work" "20-personal" "30-learning"
    "40-writing" "50-journal" "60-ideas" "99-archive")
  "The numbered area directories that make up Organicely.
Each note is its own file inside one of these (optionally in a section subdir).")

(defconst systemhalted/org-area-sections
  '(("20-personal" . ("career" "home" "health" "family" "finances"
                      "purchases" "recommendations"))
    ("30-learning" . ("ai-sessions" "articles" "books" "podcasts"
                      "courses" "videos" "subjects"))
    ("40-writing"  . ("concepts" "ideas" "drafts" "articles" "podcasts"
                      "poetry" "stories" "books" "published"))
    ("50-journal"  . ("daily" "old"))
    ("60-ideas"    . ("apps-and-tools" "business" "research-questions")))
  "Section subdirectories seeded under each area; unlisted areas stay flat.
Sections are just directories — new ones are created on demand at capture/refile.")

(defconst systemhalted/org-done-state-regexp
  (concat "^\\*+ \\(?:TODO\\|IN-PROGRESS\\|DONE\\)\\(?:[ \t]\\|$\\)"
          "\\|^[ \t]*\\(?:SCHEDULED\\|DEADLINE\\):")
  "Matches task headings or planning lines — things that belong in Reminders.")

(defun systemhalted/org-goto-or-create-heading (heading)
  "Move point onto top-level HEADING, creating it when needed.
Leaves point on the heading line itself so `org-capture' treats it as
the parent entry and appends the capture to the end of its subtree."
  (goto-char (point-min))
  (if (re-search-forward
       (format "^\\* %s[ \t]*$" (regexp-quote heading)) nil t)
      (beginning-of-line)
    (goto-char (point-max))
    (unless (bolp)
      (insert "\n"))
    (unless (bobp)
      (insert "\n"))
    (insert "* " heading "\n")
    (forward-line -1)))

(defun systemhalted/org-slug (s)
  "Kebab-case slug from S: lowercase, non-alphanumeric runs -> hyphen; keeps Unicode letters."
  (let ((x (downcase (string-trim s))))
    (setq x (replace-regexp-in-string "[^[:alnum:]]+" "-" x))
    (setq x (replace-regexp-in-string "-\\{2,\\}" "-" x))
    (setq x (string-trim x "-+" "-+"))
    (if (string-empty-p x) "note" x)))

(defun systemhalted/org-unique-file (dir stem)
  "Return a non-colliding DIR/STEM.org path, adding -N on collision."
  (let ((path (expand-file-name (concat stem ".org") dir)) (i 2))
    (while (file-exists-p path)
      (setq path (expand-file-name (format "%s-%d.org" stem i) dir))
      (setq i (1+ i)))
    path))

(defun systemhalted/org-section-dirs ()
  "Relative \"area\" and \"area/section\" directories under `org-directory'.
These are the completion candidates for capture and refile."
  (let (dirs)
    (dolist (area systemhalted/org-area-dirs)
      (push area dirs)
      (let ((adir (systemhalted/org-file area)))
        (when (file-directory-p adir)
          (dolist (sub (directory-files adir nil "\\`[^.]"))
            (when (and (file-directory-p (expand-file-name sub adir))
                       (not (string-suffix-p "-attachments" sub))
                       (not (string-suffix-p "(Attachments)" sub)))
              (push (concat area "/" sub) dirs))))))
    (nreverse dirs)))

(defun systemhalted/org-capture-file ()
  "Capture target: create a fresh note file and switch to it.
Prompts for a destination — \"Inbox\" or any existing/new \"area/section\"
directory — then a title (optional for Inbox, where the file is timestamped).
Seeds a `#+TITLE:'/`#+DATE:' header and leaves point ready for the body."
  (let* ((choice (completing-read "Capture to (Inbox, or area/section): "
                                  (cons "Inbox" (systemhalted/org-section-dirs))))
         (inbox (string-equal choice "Inbox"))
         (reldir (if inbox "00-inbox" choice))
         (dir (systemhalted/org-file reldir))
         (title (string-trim (read-string (if inbox "Title (optional): " "Title: "))))
         (stem (if (string-empty-p title)
                   (format-time-string "%Y-%m-%d-%H%M")
                 (systemhalted/org-slug title)))
         (path (systemhalted/org-unique-file dir stem)))
    (make-directory dir t)
    (set-buffer (find-file-noselect path))
    (goto-char (point-max))
    (insert "#+TITLE: " (if (string-empty-p title) stem title) "\n"
            "#+DATE: " (format-time-string "[%Y-%m-%d %a %H:%M]") "\n\n")
    (goto-char (point-max))))

(defun systemhalted/refile-note ()
  "Move the current note file (and its attachments) into a chosen area/section.
This is the directory-model replacement for `org-refile' — the weekly sweep
moves an inbox note into place by relocating its file."
  (interactive)
  (unless (and buffer-file-name
               (file-in-directory-p buffer-file-name org-directory))
    (user-error "Not visiting a note under Organicely"))
  (let* ((target-rel (completing-read "Move note to: "
                                      (systemhalted/org-section-dirs) nil nil))
         (target-dir (systemhalted/org-file target-rel))
         (dest (systemhalted/org-unique-file
                target-dir (file-name-base buffer-file-name)))
         (old buffer-file-name)
         (attach (concat (file-name-sans-extension old) "-attachments")))
    (make-directory target-dir t)
    (save-buffer)
    (rename-file old dest)
    (when (file-directory-p attach)
      (rename-file attach (concat (file-name-sans-extension dest) "-attachments")))
    (set-visited-file-name dest nil t)
    (message "Moved to %s" (file-relative-name dest org-directory))))

(defun systemhalted/journal ()
  "Open (creating if needed) today's per-day journal file for writing.
Journal entries are individual files under `50-journal/daily/' named
`YYYY-MM-DD.org'; a fresh file is seeded with a human-readable title like
\"5th June 2026, Friday\" and opened alone in book view for a distraction-free
writing session."
  (interactive)
  (let* ((day (string-to-number (format-time-string "%d")))
         (suffix (cond ((memq day '(11 12 13)) "th")
                       ((= (% day 10) 1) "st")
                       ((= (% day 10) 2) "nd")
                       ((= (% day 10) 3) "rd")
                       (t "th")))
         (title (format "%d%s %s, %s" day suffix
                        (format-time-string "%B %Y")
                        (format-time-string "%A")))
         (dir (systemhalted/org-file "50-journal/daily")))
    (make-directory dir t)
    (find-file (expand-file-name (format-time-string "%Y-%m-%d.org") dir))
    (delete-other-windows)
    (when (= (buffer-size) 0)
      (insert "#+TITLE: " title "\n"
              "#+DATE: " (format-time-string "[%Y-%m-%d %a]") "\n\n"))
    (goto-char (point-max))
    (unless (bolp) (insert "\n"))
    (unless (bound-and-true-p olivetti-mode)
      (systemhalted/book-view-toggle))))

(global-set-key (kbd "C-c J") #'systemhalted/journal)
#+end_src

** Soft guardrail: tasks belong in Reminders

The old three-file model enforced its boundaries with save-time /errors/.
A second brain doesn't need that severity — there is only one rule, and
breaking it hurts nobody but future-you. So the single remaining guard is a
=before-save-hook= that /warns/ and never blocks: if a file under
=org-directory= contains anything matching
=systemhalted/org-done-state-regexp=, saving prints a reminder that the item
has a done state and belongs in Reminders. The save itself always succeeds.

#+begin_src emacs-lisp
(defun systemhalted/org-warn-done-state ()
  "Warn (never block) when a file under `org-directory' holds a task.
Anything with TODO/IN-PROGRESS/DONE or SCHEDULED/DEADLINE has a done
state and belongs in Reminders, not in Organicely."
  (when (save-excursion
          (goto-char (point-min))
          (re-search-forward systemhalted/org-done-state-regexp nil t))
    (message "Note: this has a \"done\" state; it belongs in Reminders, not org.")))
#+end_src

** Directory seeding and per-file hook dispatch

=systemhalted/apply-org-soft-guard= runs on =org-mode-hook= and attaches the
soft done-state warning buffer-locally to any file under =org-directory=, so a
task-shaped heading gets a nudge no matter which note it lands in.

=systemhalted/ensure-org-files= creates the area and section directories up
front — the areas in =systemhalted/org-area-dirs=, each with the subdirectories
listed in =systemhalted/org-area-sections= (unlisted areas stay flat). It is
called from the main configuration's =:config= block, which runs the moment Org
first loads, so capture and refile always have real directories to complete
against — even on a machine where =~/organicely/= doesn't exist yet. There is no
per-file seeding hook any more: a note is a file, not a heading in a shared
file, so there is nothing to pre-populate when one is opened.

#+begin_src emacs-lisp
(defun systemhalted/apply-org-soft-guard ()
  "Attach the soft done-state warning to files under `org-directory'."
  (when (and buffer-file-name
             (file-in-directory-p buffer-file-name org-directory))
    (add-hook 'before-save-hook #'systemhalted/org-warn-done-state nil t)))

(defun systemhalted/ensure-org-files ()
  "Create the Organicely area and section directories if missing.
Areas listed in `systemhalted/org-area-sections' get their section subdirs;
the rest are created flat.  New sections appear on demand via capture/refile."
  (make-directory org-directory t)
  (dolist (area systemhalted/org-area-dirs)
    (let ((sections (cdr (assoc area systemhalted/org-area-sections))))
      (if sections
          (dolist (sec sections)
            (make-directory (systemhalted/org-file (concat area "/" sec)) t))
        (make-directory (systemhalted/org-file area) t)))))
#+end_src

** Source-aware inbox capture

=systemhalted/org-notes-target= is the =:function= callback for the =s=
capture template defined further below. It decides /where/ in
=00-inbox/sources.org= the new note lands based on the buffer capture was
launched from:

- If point was on a heading in any Org file other than that sources file (e.g.
  a learning note), the source heading gains an =:ID:= property (one-time, via
  =org-id-get-create=), and the note lands under a top-level heading in
  =00-inbox/sources.org= whose =:SOURCE_ID:= matches that ID. Repeated captures
  from the same source all land under the same subtree — *even if the source is
  later renamed*, because the link is by ID, not text, so a source's notes stay
  grouped for review.
- Otherwise (capture from a code buffer, plain-text scratch, or the sources
  file itself), the note falls back under the flat =* Inbox= heading created on
  demand by =systemhalted/org-goto-or-create-heading=.

Source notes are the one deliberate exception to "a note is a file": grouping
several notes about one source under a shared heading earns them a single shared
file. The matching =:PROPERTIES:= drawer is written inline rather than through
Org's property API because the heading hasn't been created at this point — we're
writing it from scratch.

#+begin_src emacs-lisp
(defun systemhalted/org-notes-target ()
  "Position point in 00-inbox/sources.org for a `file+function' capture target.
When capture is launched from an Org heading outside the inbox, ensure that
heading has an :ID: and file the new note under a top-level heading in the
inbox keyed by :SOURCE_ID:.  This groups every note about the same source
under one subtree even after the source is renamed.  When capture is launched
from anywhere else, fall back to the flat `* Inbox' heading."
  (let* ((src-buf (org-capture-get :original-buffer))
         (src-info
          (when (and src-buf (buffer-live-p src-buf))
            (with-current-buffer src-buf
              (when (and (derived-mode-p 'org-mode)
                         buffer-file-name
                         (not (systemhalted/org-buffer-file-p "00-inbox/sources.org"))
                         (ignore-errors (org-back-to-heading t) t))
                (cons (org-get-heading t t t t)
                      (org-id-get-create)))))))
    (goto-char (point-min))
    (cond
     (src-info
      (let ((title (car src-info))
            (id (cdr src-info)))
        (if (re-search-forward
             (format "^[ \t]*:SOURCE_ID:[ \t]+%s[ \t]*$" (regexp-quote id))
             nil t)
            ;; Stay on the heading line: org-capture appends the note to
            ;; the end of this subtree (point past it would misfile the
            ;; note under the next source heading).
            (org-back-to-heading t)
          (goto-char (point-max))
          (unless (bolp) (insert "\n"))
          (let ((start (point)))
            (insert "* " title "\n"
                    "  :PROPERTIES:\n"
                    "  :SOURCE_ID: " id "\n"
                    "  :END:\n")
            (goto-char start)))))
     (t
      (systemhalted/org-goto-or-create-heading "Inbox")))))
#+end_src

** Main Org configuration

The =org= =use-package= form is where everything declared above gets
wired together. The interesting clusters:

- *Global keys.* =C-c c= for capture, =C-c l= for =org-store-link=, and — in
  Org buffers — =C-c C-w= rebound to =systemhalted/refile-note= (moving a file,
  not refiling a subtree). There is deliberately no agenda binding — the agenda
  is a task view, and org holds no tasks. The mode hook turns on
  =visual-line-mode= so long paragraphs wrap softly in any Org buffer.
- *Display.* =org-startup-indented t= renders headings with virtual
  indentation; =org-hide-emphasis-markers t= suppresses the surrounding
  =/= / =*= / === characters on inline emphasis (cleaner for reading,
  still typeable); =org-ellipsis " ..."= replaces the default =...=
  folded-heading marker; =org-src-window-setup 'current-window= keeps
  =C-c '= source edits in the same window instead of splitting.
- *Babel safety.* =org-confirm-babel-evaluate t= prompts before
  executing a code block — necessary because this file itself contains
  Babel blocks that get evaluated on tangle.
- *No task vocabulary.* =org-todo-keywords= is =nil= so =C-c C-t= has
  nothing to offer. Anything with a done state goes to Reminders.

The =:config= block wires the second-brain flow:

- =org-capture-templates= has two entries. =c= Note is the everyday capture:
  its =systemhalted/org-capture-file= target asks for a destination — =Inbox= or
  any =area/section= — and a title, then creates the file. =s= Source note runs
  through =systemhalted/org-notes-target= above to group notes by their source in
  =00-inbox/sources.org=. The journal has no template on purpose — journaling
  goes through =C-c J= (=systemhalted/journal=), a writing session rather than a
  popup. Refiling is not Org's =org-refile= at all: =C-c C-w= is rebound to
  =systemhalted/refile-note=, which moves a note's file into a chosen directory,
  so there are no =org-refile-targets= to maintain.

Then the hooks: =org-mode-hook= attaches the soft done-state warning.
=systemhalted/ensure-org-files= is called directly from =:config=, which runs
the moment Org first loads (capture, refile, or opening any =.org= file) — so the
area and section directories exist before anything could target them, without
forcing Org into the startup path. =C-c a= and =C-c P= — the old
agenda and promotion keys — are
explicitly unbound so a config reload in a long-running session clears
them. Finally, Babel is loaded for Emacs Lisp, Java, Python, and shell —
Jupyter is opt-in via its own subsection below.

** Book view for reading

A book-style reading view for any buffer, inspired by Rougier's =book-mode= but
built from two small, theme-agnostic MELPA packages instead of pulling in
=nano-theme= and its global hook installations. The visual feel comes from two
things:

- =olivetti= centers the buffer body in the window with wide symmetric margins,
  giving the "page" feel.
- =mixed-pitch= renders body text in =variable-pitch= (already configured as
  Cantarell at the start of this file) while keeping =fixed-pitch= for source
  blocks, tables, inline code, and other monospace-sensitive faces. This is the
  single biggest contributor to the book look.

The other ingredients are already in place above: =org-startup-indented= for
heading indentation, =org-hide-emphasis-markers= to drop the markup characters,
and =visual-line-mode= for soft wrapping at the right margin.

Activation is on-demand via =C-c B= (capital, since =C-c b= is
=consult-buffer=). The toggle is global rather than wired into
=org-mode-hook= because not every Org buffer is a reading surface — the
inbox during a sweep is a sorting surface, where book view would harm
scanning. Journal entries, long-form drafts in =40-writing.org=, tutorial
buffers, and reading notes are where it belongs, and the keybinding makes
it trivial to invoke wherever it helps. =systemhalted/journal= (=C-c J=)
turns it on automatically for writing sessions.

#+begin_src emacs-lisp
(use-package olivetti

  :commands olivetti-mode

  (olivetti-style 'fancy))

(use-package mixed-pitch
  :ensure t
  :commands mixed-pitch-mode)

* Git and Documentation

Three file-shape packages: Magit for Git, Markdown mode for prose
documentation, YAML mode for configuration. None impose a workflow beyond
their own file type, so they cohabit in one section.

** Magit — Git through a status buffer

[[https://magit.vc/][Magit]] is the only Git interface here because it is
high-leverage and self-contained: =C-x g= opens the status buffer, every
git operation has a discoverable keybinding from there (=s= stage, =u=
unstage, =c c= commit, =b b= switch branch, =l l= log, =P p= push, =F u=
pull), and there's no global behavior change outside that buffer. No
custom hunks-on-save, no auto-commit, no smart-staging — explicit Git
remains explicit.

Commits and rebases done from inside Magit (=c c=, =r= …) open their edit
buffer right here in this Emacs and finish with =C-c C-c= — the shell's
editor never enters the picture. For Git run from an *external terminal*,
the trick is to reuse this same Emacs instead of spawning a throwaway one
that you'd have to fully quit to release =git=. That's what the always-on
=server= (see [[*Always-on minor modes and shared infrastructure]]) is for:
point Git's editor at the client with =git config --global core.editor
"emacsclient -t -a ''"= (and the matching =sequence.editor= for =rebase
-i='s todo list). The =-t= opens a terminal frame in the current terminal,
=-a ''= starts a daemon if none is reachable, and =C-c C-c= finishes /
=C-c C-k= aborts without killing Emacs. The single-key rebase commands
(=r= reword, =s= squash, =f= fixup, =p= pick, =e= edit) and the =C-c C-c=
finish key come from =with-editor= and =git-rebase=, which ride in as
Magit's own dependencies.

But those dependencies only activate once Magit itself loads, and Magit is
autoloaded — it doesn't load until the first =C-x g=. So a commit started
cold from a terminal would otherwise land in =fundamental-mode= with =C-c
C-c= unbound. =systemhalted/git-edit-buffer-setup= bridges that gap: a
=find-file-hook= that costs one string test per visited file and pulls in
=git-commit= / =git-rebase= (with their =C-c C-c= machinery) only for the
handful of files Git asks us to edit — keeping both packages out of normal
startup.

#+begin_src emacs-lisp
(defun systemhalted/git-edit-buffer-setup ()
  "Enable Magit's commit/rebase editing for buffers Git opens via emacsclient.
Magit is only autoloaded (it loads on `C-x g'), so a commit or rebase started
from an external terminal would otherwise open in `fundamental-mode' with no
`C-c C-c' to finish.  Gate on the visited file's name — a cheap per-file
string test — and load `git-commit' / `git-rebase' on demand, never at startup."
  (when buffer-file-name
    (let ((name (file-name-nondirectory buffer-file-name)))
      (cond
       ((equal name "git-rebase-todo")
        (require 'git-rebase)
        (git-rebase-mode))
       ((string-match-p "\\(?:MSG\\|_DESCRIPTION\\)\\'" name)
        (require 'git-commit)
        (git-commit-setup-check-buffer))))))

(use-package magit
  :ensure t
  :bind
  ("C-x g" . magit-status)
  :init
  (add-hook 'find-file-hook #'systemhalted/git-edit-buffer-setup))
#+end_src

** Markdown mode — README and prose docs

Three file globs: =README.md= opens in =gfm-mode= (GitHub-flavored Markdown,
which renders =\n= as line breaks and supports task-list checkboxes) since
that's where the README convention lives; every other =.md= or =.markdown=
file uses plain =markdown-mode=. The GFM split matters because writing
README bullets that render correctly on GitHub requires the GFM-specific
parsing.

Pipe tables are the sore spot. =markdown-mode= re-aligns a table *while you
edit it* (=markdown-table-align-p= is =t= by default), but a table that was
pasted or generated elsewhere — a blog post table, say — stays ragged in the
buffer until you happen to touch a cell, so the columns don't line up and
it's hard to read. =systemhalted/markdown-align-all-tables= closes that gap:
it walks the buffer and runs =markdown-table-align= on every table. The
=markdown-mode= hook wires it in at both ends — it runs once *when a file is
visited*, so a freshly opened buffer shows neat columns straight away, and
again on a buffer-local =before-save-hook=, so the aligned form is what lands
on disk. The visit-time pass binds =buffer-undo-list= to =t= and restores the
buffer's modified flag afterward, so opening a file neither dirties the buffer
nor writes anything — the on-disk file is only rewritten by a *real* save.
=markdown-table-align= is still there to align a single table on demand.
Alignment is cosmetic whitespace only — Kramdown/GFM ignore it — so it never
changes how the table renders on the published site.

For proportional fonts or mixed-width table content, =valign-mode= provides a
visual overlay that keeps table columns lined up in the buffer without changing
the file contents. It is enabled only in Markdown buffers.

#+begin_src emacs-lisp
(defun systemhalted/markdown-align-all-tables ()
  "Re-align every pipe table in the current Markdown buffer.
`markdown-mode' realigns a table as you edit it (`markdown-table-align-p'
is t), but a pasted or generated table stays ragged until touched.  Walk
the buffer and run `markdown-table-align' on each table so the columns
line up.  Used on `before-save-hook'; also callable interactively."
  (interactive)
  (when (derived-mode-p 'markdown-mode)
    (save-excursion
      (goto-char (point-min))
      (while (re-search-forward "^[ \t]*|" nil t)
        (if (markdown-table-at-point-p)
            (progn
              (markdown-table-align)
              (goto-char (markdown-table-end)))
          (forward-line 1))))))

(use-package markdown-mode
  :ensure t
  :mode
  (("README\\.md\\'" . gfm-mode)
   ("\\.md\\'" . markdown-mode)
   ("\\.markdown\\'" . markdown-mode))
  :hook
  (markdown-mode . (lambda ()
                     (add-hook 'before-save-hook
                               #'systemhalted/markdown-align-all-tables nil t)
                     ;; Align on visit too, so a freshly opened file shows neat
                     ;; columns without waiting for a save.  Suppress undo and
                     ;; keep the buffer's modified state so opening writes
                     ;; nothing and leaves nothing to undo.
                     (let ((was-modified (buffer-modified-p))
                           (buffer-undo-list t))
                       (systemhalted/markdown-align-all-tables)
                       (unless was-modified (set-buffer-modified-p nil))))))

(use-package valign
  :ensure t
  :hook
  (markdown-mode . valign-mode))
#+end_src

* Programming

Language intelligence is centered on =lsp-mode=: one protocol, one
completion path through Corfu via =capf=, one diagnostics surface through
Flycheck. =lsp-mode= owns the protocol, =lsp-ui= layers a few minimal
in-buffer affordances, =lsp-java= drives JDT LS, =lsp-pyright= drives
Pyright, =rustic= drives =rust-analyzer=, and =dap-mode= reuses JDT LS
for breakpoints in Java. The
subsections below introduce each piece next to its rationale; a usage
tutorial for the everyday LSP/DAP commands lives at the end of this

This replaces older language-specific stacks (Elpy, Tide, company-*) — see

** Process I/O and shell environment

Two startup-time settings that the rest of the section depends on:

- =read-process-output-max= controls the largest chunk Emacs reads from a
  subprocess in one go. The default (4 KiB on most systems) is fine for
  =grep= but pathological for =lsp-mode=, which streams large JSON
  responses from JDT LS and Pyright. Raising it to 1 MiB cuts the
  per-response syscall count by ~250×, which is the single biggest
  responsiveness win for Java buffers.
- =exec-path-from-shell= imports =PATH=, =MANPATH=, =JAVA_HOME=, and
  =SDKMAN_EL_DIR= from a login + interactive shell on startup. GUI Emacs
  (macOS, Linux Wayland via PGTK, Linux X11 launched from outside a
  terminal) does *not* inherit the interactive shell environment, so
  SDKMAN / Homebrew / MacPorts / =update-alternatives= entries are
  otherwise invisible to the Emacs process. =SDKMAN_EL_DIR= is imported
  for the same reason: it points the SDKMAN section below at a local
  =sdkman.el= checkout, and without this import a GUI Emacs would never
  see it and would always fall back to the GitHub install. SDKMAN's init only runs in interactive shells
  (=~/.zshrc=), not login shells (=~/.zprofile=), which is why the
  arguments are =-l -i= — both flags are needed to source the full
  shell startup. The =:if= guard skips this in headless batch runs where
  there's no shell to spawn.

#+begin_src emacs-lisp
(setq read-process-output-max (* 1024 1024))

(use-package exec-path-from-shell
  :ensure t
  :if (or (daemonp) (memq window-system '(mac ns x pgtk)))
  :init
  (setq exec-path-from-shell-variables '("PATH" "MANPATH" "JAVA_HOME" "SDKMAN_EL_DIR"))
  ;; SDKMAN init lives in ~/.zshrc (interactive), not ~/.zprofile (login),
  ;; so a plain login shell sees no JAVA_HOME. Use -l -i to source both.
  (setq exec-path-from-shell-arguments '("-l" "-i"))
  :config
  (exec-path-from-shell-initialize))
#+end_src

** SDKMAN — project-local JDK selection

JDK locations are *not* hard-coded. The =sdkman.el= package ties Java
projects to whichever JDK their =.sdkmanrc= names. Loading is hybrid and
driven by the =SDKMAN_EL_DIR= environment variable: on a development
machine =SDKMAN_EL_DIR= points at a local checkout (wherever it happens
to live), and that directory is put on =load-path= directly so edits
take effect on reload. When the variable is unset or points nowhere
real, =package-vc= installs the package from
=github.com/systemhalted/sdkman.el= instead. One committed config works
on every machine; the only per-machine choice is whether to export
=SDKMAN_EL_DIR=. (Because GUI Emacs doesn't inherit the shell
environment, =exec-path-from-shell= imports =SDKMAN_EL_DIR= alongside
=PATH= — see above.) When a Java buffer is visited,
=global-sdkman-mode= reads the nearest =.sdkmanrc= and applies the listed
SDKMAN candidates (=java=, =maven=, =gradle=, …) to the buffer's
=process-environment=, =exec-path=, and =PATH=; sets =JAVA_HOME= /
=MAVEN_HOME= / =GRADLE_HOME= as appropriate; and points
=lsp-java-java-path= at that JDK's =bin/java= so JDT LS itself launches
with the project's JDK rather than whatever SDKMAN considers default
globally. The Java language level still comes from the project model
(=maven.compiler.release=, Gradle toolchain config, Eclipse metadata),
not =JAVA_HOME= alone.

=systemhalted/lsp-deferred-unless-excluded= is the helper that
=lsp-mode='s =java-mode= hook calls instead of plain =lsp-deferred=. It
defers to =sdkman-lsp-java-excluded-file-p= to skip generated JDT LS
support files (workspace cache, decompiled server source) — opening one
of those for inspection shouldn't spin up a fresh JDT LS session.
=lsp-deferred= itself waits for the buffer to become visible, so by the
time JDT LS starts, the SDKMAN buffer-local state applied via
=after-change-major-mode-hook= is already in place.

#+begin_src emacs-lisp
(defvar systemhalted/sdkman-local-checkout
  (let ((dir (getenv "SDKMAN_EL_DIR")))
    (and dir (file-directory-p dir) (file-name-as-directory dir)))
  "Local sdkman.el checkout from $SDKMAN_EL_DIR, or nil to install from GitHub.")

(if systemhalted/sdkman-local-checkout
    ;; Development machine: $SDKMAN_EL_DIR points at the live checkout, so load
    ;; it directly and edits take effect on the next reload.
    (use-package sdkman
      :preface (add-to-list 'load-path systemhalted/sdkman-local-checkout)
      :commands (sdkman-mode global-sdkman-mode sdkman-lsp-java-excluded-file-p)
      :init (global-sdkman-mode 1))
  ;; No local checkout: install from GitHub. :rev :newest tracks the latest
  ;; commit (use-package-vc-prefer-newest is nil by default, which would pin to
  ;; a release tag).
  (use-package sdkman
    :vc (:url "https://github.com/systemhalted/sdkman.el" :rev :newest)
    :commands (sdkman-mode global-sdkman-mode sdkman-lsp-java-excluded-file-p)
    :init (global-sdkman-mode 1)))

** LSP-mode — protocol, completion, memory tuning

[[https://emacs-lsp.github.io/lsp-mode/][lsp-mode]] is the protocol layer.
It attaches to Java via =systemhalted/lsp-deferred-unless-excluded=
(SDKMAN-aware) and to JS/TS/web/CSS/Rust via plain =lsp-deferred=. The
non-default settings exist for two reasons:

*Memory bounds.* A Java editing session previously pushed Emacs above
50 GiB of resident memory; JDT LS itself is capped by =lsp-java-vmargs=,
so the runaway growth came from Emacs retaining protocol messages, file
watches, hover payloads, and completion metadata indefinitely:

- =lsp-enable-file-watchers nil= and =lsp-file-watch-threshold 2000= stop
  Emacs from registering recursive watchers on huge monorepos. JDT LS
  has its own file watching; doing it again in Emacs is duplicated work
  and unbounded notification noise.
- =lsp-log-io nil= and =lsp-log-max 2000= keep the IO log tiny.
  =lsp-io-messages-max 200= caps the kept-in-memory protocol history.
- =lsp-keep-workspace-alive nil= shuts a workspace down when its last
  buffer is killed, releasing the JDT LS process and its protocol state
  immediately rather than at next GC.

*Completion provider, the =:none= gotcha.* =lsp-completion-provider= is
set to =:none=, *not* =:capf=, even though completion is wanted. The
naming is misleading: =lsp-completion-mode= always registers
=lsp-completion-at-point= on =completion-at-point-functions= regardless
of this setting; the provider option only controls whether =lsp-mode=
tries to auto-enable =company-mode=. With Corfu (capf-based) and no
company installed, =:capf= still falls through to the company branch and
warns "Unable to autoconfigure company-mode" once per LSP buffer.
=:none= skips that branch entirely while leaving the capf wired up,
which is what Corfu consumes.

*Quiet UI by default.* =lsp-enable-symbol-highlighting nil=,
=lsp-headerline-breadcrumb-enable nil=, =lsp-modeline-code-actions-enable nil=,
=lsp-enable-on-type-formatting nil=, and the
=lsp-completion-show-detail/-kind nil= pair drop the chatty defaults.
Hover docs, code actions, symbol highlight, and on-type formatting are
all available on demand via the bindings below; they're just not on
constantly.

The bindings hang off the standard =C-c l= prefix. =C-c C-f= is the
format-buffer chord, =h h= hover docs, =i= toggles =corfu-popupinfo= for
completion-side docs, =H= / =F= toggle the noisier features above on a
per-session basis, =R= restarts the workspace, and =j u= refreshes the
Java project model when the build file changes.

#+begin_src emacs-lisp
(use-package lsp-mode

  :bind
  (:map lsp-mode-map
        ("C-c C-f" . lsp-format-buffer)
        ("C-c l h h" . lsp-describe-thing-at-point)
        ("C-c l i" . corfu-popupinfo-toggle)
        ("C-c l H" . lsp-toggle-symbol-highlight)

        ("C-c l R" . lsp-workspace-restart)
        ("C-c l j u" . lsp-java-update-project-configuration))
  :custom
  (lsp-keymap-prefix "C-c l")
  ;; :none is the Corfu-correct value despite the name. lsp-completion-mode
  ;; registers `lsp-completion-at-point' on `completion-at-point-functions'
  ;; unconditionally; the provider setting only controls whether lsp-mode
  ;; tries to auto-enable company-mode. With Corfu we don't have company,
  ;; so :capf still takes the company branch and warns "Unable to
  ;; autoconfigure company-mode." :none skips that branch entirely while
  ;; leaving capf wired up — which is what Corfu consumes.

  (lsp-completion-show-kind nil)
  (lsp-enable-file-watchers nil)
  (lsp-enable-symbol-highlighting nil)
  (lsp-enable-on-type-formatting nil)
  (lsp-idle-delay 1.0)
  (lsp-io-messages-max 200)
  (lsp-keep-workspace-alive nil)
  (lsp-log-io nil)
  (lsp-log-max 2000)
  (lsp-file-watch-threshold 2000)
  (lsp-headerline-breadcrumb-enable nil)

** DAP-mode — Java debugging through JDT LS

=dap-mode= is the Emacs Debug Adapter Protocol front end. Loading
=dap-java= in =:config= wires it through the JDT LS that =lsp-java=
already installs — no extra server. =C-c l d= opens the DAP hydra
(stepping, breakpoints, watch). =dap-auto-configure-features= keeps the
default four panes (sessions, locals, controls, tooltip); the others
(REPL, expressions, breakpoints list) are available but not on by
default.

=dap-java--populate-default-args= resolves the main class, classpath,
and debug-server metadata through synchronous JDT LS commands. Cold
Maven imports can exceed the normal interactive LSP timeout (30 s) even
when the server is healthy, so =systemhalted/with-longer-dap-java-timeout=
wraps that call with a 60 s =lsp-response-timeout=. =advice-remove=
followed by =advice-add= makes the install idempotent under config
reloads — applying the advice twice would otherwise double-wrap and
inflate the timeout to 120 s.

JPMS-modular Maven modules need a second fix. When a module declares
itself in =module-info.java= (e.g. =module kisoku.api=), JDT LS's
=vscode.java.resolveMainClass= returns the *module-qualified* main class
=kisoku.api/in.systemhalted.kisoku.api.Test=. Upstream
=dap-java--populate-launch-args= calls =vscode.java.resolveClasspath=,
which returns a two-element sequence =[modulePaths classPaths]=, but keeps
only =classPaths= (=cl-second=) and pins =:modulePaths= to an empty
vector. A module-qualified class launched in classpath mode (=java -cp …
kisoku.api/in.systemhalted.kisoku.api.Test=) makes the JVM rewrite =/= to
=.=, yielding =ClassNotFoundException= and a failed jdwp handshake.
=systemhalted/dap-java-add-module-paths= advises the launch populator with
=:filter-return= — the cleanest hook because that function returns the
finished conf plist — re-resolves the classpath, and when =modulePaths=
(=cl-first=) is non-empty sets =:modulePaths= forcefully. It must use
=plist-put=, not =dap--put-if-absent=: the template's pre-set empty vector
is non-nil, so =dap--put-if-absent= (which only fills =nil= slots) would
never replace it. With real module paths in place the bundled java-debug
adapter assembles =--module-path … -m module/Class= itself. For
non-modular projects =modulePaths= comes back empty and the conf is
returned untouched, so ordinary classpath launches are unaffected. Use the
plain *Java Run Configuration* template for modular modules; the
*compile/attach* variant hand-rolls its own =java -cp … <mainClass>=
command and is not module-aware. This is a workaround for an upstream
=lsp-java= bug; remove the advice once it is fixed there.

#+begin_src emacs-lisp
(defun systemhalted/with-longer-dap-java-timeout (orig-fun &rest args)
  "Call ORIG-FUN with a longer timeout for DAP Java launch metadata."
  ;; `dap-java-debug' resolves main class, classpath, and debug-server
  ;; metadata through synchronous JDT LS commands. Cold Maven imports can
  ;; exceed the normal interactive LSP timeout without implying server failure.
  (let ((lsp-response-timeout 60))
    (apply orig-fun args)))

(defun systemhalted/dap-java-add-module-paths (conf)
  "Populate :modulePaths in CONF for JPMS-modular Java launch configs.

`dap-java--populate-launch-args' resolves the classpath via
`vscode.java.resolveClasspath', which returns a two-element sequence
[MODULE-PATHS CLASS-PATHS].  Upstream keeps only CLASS-PATHS (cl-second)
and forces :modulePaths to an empty vector, so a module declared in a
`module-info.java' is launched in classpath mode with a module-qualified
main class (e.g. \"kisoku.api/in.systemhalted.kisoku.api.Test\").  The JVM
then rewrites \"/\" to \".\" and fails with ClassNotFoundException before
the jdwp handshake.

This filter re-resolves the classpath, and when MODULE-PATHS (cl-first)
is non-empty sets :modulePaths forcefully (plist-put, not
`dap--put-if-absent', because the template's pre-set empty vector is
non-nil and would never be overridden).  The bundled java-debug adapter
then assembles `--module-path ... -m module/Class' on its own.  For
non-modular projects MODULE-PATHS is empty and CONF is returned
unchanged.

Workaround for upstream emacs-lsp/lsp-java: dap-java--populate-launch-args
discards resolveClasspath's module paths (keeps only cl-second).  Remove
this advice once that is fixed upstream.  See
https://github.com/emacs-lsp/lsp-java/issues/511."
  (-let [(&plist :mainClass main-class :projectName project-name) conf]
    (when (and main-class project-name)
      (let* ((resolved (with-lsp-workspace (lsp-find-workspace 'jdtls)
                         (lsp-send-execute-command
                          "vscode.java.resolveClasspath"
                          (vector main-class project-name))))
             (module-paths (and resolved (elt resolved 0))))
        (when (and module-paths (not (seq-empty-p module-paths)))
          (setq conf (plist-put conf :modulePaths module-paths))))))
  conf)

(use-package dap-mode
  :ensure t
  :after lsp-java
  :bind
  (:map lsp-mode-map
        ("C-c l d" . dap-hydra))
  :custom
  (dap-auto-configure-features '(sessions locals controls tooltip))
  :config
  (require 'dap-java)
  (advice-remove 'dap-java--populate-default-args
                 #'systemhalted/with-longer-dap-java-timeout)
  (advice-add 'dap-java--populate-default-args
              :around
              #'systemhalted/with-longer-dap-java-timeout)
  (advice-remove 'dap-java--populate-launch-args
                 #'systemhalted/dap-java-add-module-paths)
  (advice-add 'dap-java--populate-launch-args
              :filter-return
              #'systemhalted/dap-java-add-module-paths))
#+end_src

** Rust — rustic and rust-analyzer

=rustic= is the major mode for =.rs= files — a cargo-integrated superset
of =rust-mode= that adds the =C-c C-c= cargo bindings (run / test / build /
clippy), a colorized compilation buffer, and rustfmt-on-save. Language
intelligence comes from =lsp-mode='s built-in =rust-analyzer= client, so —
exactly like =ts-ls= for TypeScript — there is *no separate lsp package* to
install; the client ships with =lsp-mode=.

=rustic= can start lsp on its own, but here that auto-setup is disabled
(=rustic-lsp-client nil=) and =rustic-mode= is attached through =lsp-mode='s
own =:hook= group with plain =lsp-deferred= (see the LSP-mode section above),
the same way every other language is wired. Keeping all lsp-managed modes in
that one list makes the stack discoverable, and it guarantees a single
=rust-analyzer= process per workspace rather than one from rustic plus one
from the shared hook.

=lsp-rust-analyzer-cargo-watch-command "clippy"= makes rust-analyzer run
*clippy* instead of plain =cargo check=, so clippy's richer idiomatic-Rust
lints surface in Flycheck alongside ordinary type errors — worth the slightly
slower check while learning the language. =rustic-format-trigger 'on-save=
runs rustfmt over the buffer just before every save.

The toolchain must be on the =PATH= =exec-path-from-shell= imports.
=rustup= is the portable install (it drops =rustc=, =cargo=, and
=rust-analyzer= under =~/.cargo/bin=, which must be on your shell =PATH=):

#+begin_src sh
# Portable: rustup installs rustc + cargo, then add the analyzer component.
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
rustup component add rust-analyzer   # rustfmt + clippy ship in the default profile
#+end_src

Distro packages also work: macOS =sudo port install rust rust-analyzer=
(MacPorts) or =brew install rustup rust-analyzer= then =rustup-init=
(Homebrew); Fedora =dnf install rust cargo rust-analyzer rustfmt clippy=;
Ubuntu =apt install rustup= then =rustup component add rust-analyzer=
(Ubuntu's standalone =rust= packages lag — rustup is preferred).

#+begin_src emacs-lisp
(use-package rustic
  :ensure t
  :mode ("\\.rs\\'" . rustic-mode)
  :custom
  ;; Don't let rustic auto-start lsp; rustic-mode is hooked via lsp-mode's own
  ;; :hook list like every other language, so all lsp-managed modes live in one
  ;; place — and only one rust-analyzer starts per workspace.
  (rustic-lsp-client nil)
  ;; The format-on-save knob is `rustic-format-trigger', set to the symbol
  ;; 'on-save (there is no `rustic-format-on-save'); rustfmt runs over the
  ;; buffer just before each save via `rustic-format-on-save-method'.
  (rustic-format-trigger 'on-save)
  ;; rust-analyzer runs clippy (not plain cargo check) so its richer lints
  ;; surface in Flycheck — valuable while learning idiomatic Rust. We set this
  ;; directly because rustic's own lsp setup (which would force it back to
  ;; "check") is disabled via `rustic-lsp-client nil' above.
  (lsp-rust-analyzer-cargo-watch-command "clippy"))
#+end_src

** Web editing — one mode per file type

The web stack is intentionally split by file extension rather than
collapsed onto a single web-or-JS mega-mode. Each mode owns the file
types it's actually good at:

- =web-mode= handles =.html= and =.tsx= — markup-heavy templates where
  the JSX-in-TS embedding needs the same multi-language-block awareness
  as templated HTML. Indent offsets are all locked to 2.
- =rjsx-mode= handles =.jsx= and =.js= — JSX-aware editing built on
  =js2-mode=. =js2-mode-show-parse-errors nil= silences the on-by-default
  inline error highlighting; Flycheck is the diagnostics surface for the
  whole stack, and =js2-mode='s parser is too noisy on modern JS that
  uses optional chaining, decorators, etc.
- =typescript-mode= handles =.ts= — plain TS, no JSX. =.tsx= goes to
  =web-mode= above because the JSX-in-TS combo benefits from web-mode's
  block awareness more than typescript-mode's stricter TS grammar.
- =css-mode= is built-in and configured just to set =css-indent-offset=
  to 2 and route through =lsp-deferred= so the language server handles
  diagnostics and navigation.

All four feed into =lsp-mode='s hook (the =js-mode rjsx-mode
typescript-mode web-mode css-mode= group up above), so jump-to-def,
rename, and completion work uniformly across the stack.

The JS/TS/TSX buffers are served by =lsp-mode='s =ts-ls= client, which
— like Pyright above — expects its binary on the =PATH= Emacs
inherits: =typescript-language-server=, installed via npm. The global
=typescript= alongside it is only a fallback; the server prefers a
project's own =node_modules/typescript=, so per-project compiler
versions still win. npm itself comes from MacPorts or Homebrew on
macOS, =dnf install nodejs= on Fedora, =apt install npm= on Ubuntu.

#+begin_src sh
npm install -g typescript-language-server typescript
#+end_src

code detected." (7027) being the loudest — and they land in Flycheck looking

** Shell scripts and dotfiles

Shell scripts and shell dotfiles use the built-in =sh-mode=. =lsp-mode=
ships a client for =bash-language-server= (no wrapper package needed) and
Flycheck routes diagnostics through =shellcheck= when it's on =PATH=.
Both are external; install before opening a shell file:

#+begin_src sh
# bash-language-server (npm, all platforms):
npm install -g bash-language-server

# shellcheck:
# macOS, MacPorts:   sudo port install shellcheck
# macOS, Homebrew:   brew install shellcheck
# Fedora:            sudo dnf install ShellCheck
# Ubuntu / Debian:   sudo apt install shellcheck
#+end_src

=sh-mode= ships with shebang and =$SHELL= detection that handles ordinary
=.sh= files correctly, so this section only teaches it about the
extensionless zsh dotfiles (=.zshrc=, =.zshenv=, =.zprofile=, =.zlogin=,
=.zlogout=) — files where the name alone is the only hint to the dialect.
=systemhalted/sh-pick-shell= then runs on =sh-mode-hook=, matches against
a tight regex of known names, and calls =sh-set-shell= with =zsh= for
those plus any =*.zsh= file, or =bash= for =.bashrc= / =.bash_profile= /
=.bash_login= / =.bash_logout= / =.bash_aliases=. Ambiguous names like
=.profile= or =foo.zshrc= (which could be a user's own template) are left
alone so =sh-mode='s shebang/=$SHELL= detection still gets a chance.

#+begin_src emacs-lisp
(use-package sh-script
  :ensure nil
  :mode (("\\.zshrc\\'"    . sh-mode)
         ("\\.zshenv\\'"   . sh-mode)
         ("\\.zprofile\\'" . sh-mode)
         ("\\.zlogin\\'"   . sh-mode)
         ("\\.zlogout\\'"  . sh-mode))
  :hook
  (sh-mode . systemhalted/sh-pick-shell)
  (sh-mode . lsp-deferred)
  :config
  (defun systemhalted/sh-pick-shell ()
    "Set `sh-shell' for known shell dotfiles; leave the default alone otherwise.
Matching only well-known names avoids clobbering `sh-mode''s own detection
\(shebang, `$SHELL') for ambiguous files like `.profile' or `foo.zshrc'."
    (when buffer-file-name
      (let ((name (file-name-nondirectory buffer-file-name)))
        (cond
         ((or (string-match-p "\\`\\.?z\\(?:shrc\\|shenv\\|profile\\|login\\|logout\\)\\'" name)
              (string-match-p "\\.zsh\\'" name))
          (sh-set-shell "zsh" nil nil))
         ((string-match-p "\\`\\.?bash\\(?:rc\\|_profile\\|_login\\|_logout\\|_aliases\\)\\'" name)
          (sh-set-shell "bash" nil nil)))))))
#+end_src

** Using LSP and DAP

This sub-section is documentation, not configuration; nothing here tangles
into =systemhalted.el=. It captures the day-to-day workflow for the LSP and
DAP stacks set up above so the keys and commands live next to the code that
binds them.

*** LSP — everyday code intelligence

LSP attaches automatically to =java-mode=, =python-mode=, =js-mode=,
=rjsx-mode=, =typescript-mode=, =web-mode=, =css-mode=, =rustic-mode=, and
=sh-mode= via the =lsp-deferred= hook on =lsp-mode=. The mode line shows
=LSP[<server>]= once the server is up (=jdtls= for Java, =pyright= for
Python, =rust-analyzer= for Rust, etc.). If it does not appear, the server
probably failed to start — check =*lsp-log*=.

Jumping to a definition (the most common move, e.g. from a Java method call
to its declaration): place point on the symbol and press =M-.=. =lsp-mode=
installs an =xref= backend when it attaches, so the standard Emacs key routes
through the language server automatically — for Java, that is JDT LS. =M-,=
returns to where you jumped from; the stack unwinds cleanly across repeated
jumps. =C-c l g g= is the =lsp-mode=-native equivalent and is useful when you
want LSP semantics specifically.

The LSP prefix is =C-c l=. The most-used commands:

| Keys        | Command                  | What it does                          |
|-------------+--------------------------+---------------------------------------|
| =M-.=       | =xref-find-definitions=  | jump to definition (routed via LSP)   |
| =M-,=       | =xref-go-back=           | return to the previous location       |
| =C-c l g g= | =lsp-find-definition=    | jump to definition (LSP-native)       |
| =C-c l g r= | =lsp-find-references=    | list references                       |
| =C-c l g i= | =lsp-find-implementation=| jump to implementation                |
| =C-c l r r= | =lsp-rename=             | rename symbol across the workspace    |
| =C-c l r o= | =lsp-organize-imports=   | sort imports, drop unused ones        |
| =C-c l a a= | =lsp-execute-code-action=| quick fixes, imports, refactors       |
| =C-c l h h= | =lsp-describe-thing-at-point= | hover docs                       |
| =C-c l i=   | =corfu-popupinfo-toggle= | toggle completion docs for buffer     |
| =C-c l H=   | =lsp-toggle-symbol-highlight= | toggle symbol highlighting       |
| =C-c l F=   | =lsp-toggle-on-type-formatting= | toggle on-type formatting       |
| =C-c l R=   | =lsp-workspace-restart= | restart current LSP workspace          |
| =C-c l j u= | =lsp-java-update-project-configuration= | refresh Java project model |
| =C-c C-f=   | =lsp-format-buffer=      | format the buffer                     |

=which-key= is enabled, so pressing =C-c l= and pausing pops up the full
submenu — useful for discovering the rest of the prefix without memorizing
it.

Companion behaviors that are not LSP commands but feel like them:

- Symbol documentation is manual, not hover-driven: =C-c l h h= asks the
  language server about the symbol at point.
- Completion documentation is manual: =C-c l i= toggles the Corfu docs panel
  for the current buffer.
- Flycheck owns diagnostics: =C-c ! n= / =C-c ! p= step through errors,
  =C-c ! l= lists them.
- Corfu handles in-buffer completion after three typed characters and a short
  pause: =TAB= accepts, =M-n= / =M-p= cycle.
- Yasnippet expands snippets at point with =TAB=.
- Symbol highlighting and on-type formatting are off by default. Use
  =C-c l H= to toggle symbol highlighting and =C-c l F= to toggle on-type
  formatting when you want them temporarily. Full-buffer formatting remains
  available at =C-c C-f=.

When something is off, the two recovery commands worth memorizing:

- =C-c l R= — restart the server for the current project.
- =M-x lsp-describe-session= — see which servers are running and where.

For Java specifically, JDT LS imports the project on first open. Wait for
=LSP[jdtls]= in the mode line before invoking refactors or debugging — code
actions and =dap-java-debug-*= require the project model to be ready.
Automatic build-configuration updates are interactive; after changing
=pom.xml=, =build.gradle=, or dependency state outside Emacs, use =C-c l j u=
to ask JDT LS to refresh the project model.

*** TypeScript and JavaScript — auto-imports and project setup

The JS/TS/TSX stack is served by =ts-ls= (the =typescript-language-server=
binary on =PATH= — see the install note in the Web editing section). The
first time a file from a new project is opened, =lsp-mode= prompts for the
project root: answer =i= to import the suggested root (usually the git
root). The choice is remembered in =.lsp-session-v1=; get it wrong and
=M-x lsp-workspace-folders-remove= undoes it. The mode line shows
=LSP[ts-ls]= once the server is up.

Auto-import works two ways:

1. /Completion-driven./ Type the symbol name and accept a Corfu candidate
   that comes from another module — =ts-ls= attaches the =import= statement
   as an additional edit, and =lsp-mode= inserts it at the top of the file
   on accept. The completion popup deliberately hides the detail column
   (=lsp-completion-show-detail nil=), so the source module is not shown
   inline; toggle the Corfu docs panel with =C-c l i= when it matters which
   module a candidate imports from.
2. /Code-action-driven./ For a symbol already in the buffer that Flycheck
   flags with "Cannot find name ...": put point on it and press =C-c l a a=,
   then pick the "Add import from ..." action. This is the route for pasted
   code.

Housekeeping around imports:

- =C-c l r o= (=lsp-organize-imports=) sorts the import block and removes
  unused entries. It is manual by design — nothing in the config runs it on
  save.
- =C-c l r r= renames a symbol across every file the project's
  =tsconfig.json= covers, imports included.

Per-file compiler settings come from the nearest =tsconfig.json= above the
file, resolved by =tsserver= itself. A project with several tsconfigs
(say, root + =tsconfig.server.json= + =client/tsconfig.json=) needs no
extra Emacs configuration — each buffer is checked against the config that
governs it. The server also prefers the project's own
=node_modules/typescript= over the globally installed compiler, so
diagnostics match what =tsc= says in that repo.

*** Rust — cargo commands and rust-analyzer assists

Rust buffers are =rustic-mode= driven by =rust-analyzer= (the binary on
=PATH= — see the install note in the Rust section above). The first time a
file from a new crate is opened, =lsp-mode= prompts for the project root:
answer =i= to import the suggested root — pick the directory holding the
workspace =Cargo.toml=, so rust-analyzer indexes the whole Cargo workspace
rather than a single crate. The mode line shows =LSP[rust-analyzer]= once it
is up; on a large workspace the first index takes a moment, and code actions
are unavailable until it finishes.

Diagnostics come from *clippy*, not plain =cargo check= (the config sets
=lsp-rust-analyzer-cargo-watch-command "clippy"=), so clippy's idiomatic-Rust
lints land in Flycheck alongside type errors. Step through them with the same
=C-c ! n= / =C-c ! p= / =C-c ! l= as every other language.

*Generating code that doesn't exist yet* is rust-analyzer's most useful
assist and runs through the standard =C-c l a a= (=lsp-execute-code-action=):

- Call a method or function you have not written, e.g. =doc.reload();=.
  Flycheck flags it; put point on the call and press =C-c l a a=, then choose
  /"Generate a method `reload`"/ (or /"Generate function"/ for a free
  function). rust-analyzer inserts a stub in the right =impl= block.
- On an =impl Trait for T= header, =C-c l a a= → /"Implement missing
  members"/ stubs out every required trait method.
- The same menu offers /"Generate `new`"/, getters/setters, and
  /"Import ..."/ for an unresolved path — the Rust analogue of the
  TypeScript auto-import flow above.

Formatting is automatic: =rustic-format-trigger 'on-save= runs rustfmt over
the buffer just before each save, so there is rarely a need to format by
hand (=C-c C-f= still does it on demand).

Beyond LSP, =rustic= adds a cargo command layer on the =C-c C-c= prefix
(=C-c C-p= opens =rustic-popup=, a transient that lists them all):

| Keys          | Command                  | What it does                  |
|---------------+--------------------------+-------------------------------|
| =C-c C-c C-r= | =rustic-cargo-run=       | cargo run                     |
| =C-c C-c C-t= | =rustic-cargo-test=      | cargo test (whole crate)      |
| =C-c C-c C-c= | =rustic-cargo-current-test= | run the test at point      |
| =C-c C-c C-b= | =rustic-cargo-build=     | cargo build                   |
| =C-c C-c C-k= | =rustic-cargo-check=     | cargo check                   |
| =C-c C-c C-l= | =rustic-cargo-clippy=    | cargo clippy                  |
| =C-c C-c C-f= | =rustic-cargo-fmt=       | cargo fmt (whole project)     |
| =C-c C-c C-u= | =rustic-compile=         | run an arbitrary cargo/shell command |
| =C-c C-c a=   | =rustic-cargo-add=       | cargo add a dependency        |
| =C-c C-p=     | =rustic-popup=           | transient menu of all of the above |

For the =geditrs= workspace specifically, =rustic-cargo-run= / =-test=
operate on the crate owning the current buffer; to target one workspace
member explicitly (e.g. =cargo run -p tui=), use =C-c C-c C-u=
(=rustic-compile=) and type the full command — the compilation buffer is
colorized and its file:line references are clickable.

*** DAP — debugging Java

DAP turns =lsp-mode= buffers into debugger front-ends, but it is not enabled
for every LSP buffer at startup. That keeps normal Java editing lighter and
avoids allocating debugger UI state until it is needed. =C-c l d= opens
=dap-hydra=, the transient menu that exposes every debug command without
having to memorize bindings.

A typical first session in a Java project:

1. Open a source file. Wait for =LSP[jdtls]= in the mode line.
2. Move point to the line you want to pause on and run
   =M-x dap-breakpoint-toggle= (or =b t= from the hydra). A red dot appears
   in the fringe.
3. Launch one of:
   - =M-x dap-java-debug= — debug a class with a =main= method.
   - =M-x dap-java-debug-test-method= — debug the JUnit test at point.
   - =M-x dap-java-debug-test-class= — debug every test in the current class.
4. Execution stops at the breakpoint. The /Locals/, /Sessions/, and
   /Controls/ side windows open automatically (from
   =dap-auto-configure-features=).

Stepping through, mostly via =dap-hydra=:

| Hydra key | Command           | What it does           |
|-----------+-------------------+------------------------|
| =n=       | =dap-next=        | step over              |
| =i=       | =dap-step-in=     | step into              |
| =o=       | =dap-step-out=    | step out               |
| =c=       | =dap-continue=    | continue               |
| =r=       | =dap-restart-frame= | restart current frame |
| =q=       | =dap-disconnect=  | end the session        |

Other commands worth knowing:

- =M-x dap-ui-inspect-thing-at-point= — inspect the value at point. Hovering
  also shows a tooltip.
- =M-x dap-breakpoint-condition= — turn the breakpoint on the current line
  into a conditional one (Java boolean expression).
- =M-x dap-breakpoint-log-message= — make a breakpoint log instead of
  pause; useful for temporary tracing without recompiling.
- =M-x dap-debug-last= — re-run the most recent launch configuration.

Breakpoints persist per project in =.dap-breakpoints=, which is already
gitignored.

*** Common failure modes

- /JDT LS does not start./ Check =*lsp-log*=. If
  =eclipse.jdt.ls/server/= under =~/.emacs.d/= is empty, run
  =M-x lsp-install-server RET jdtls=.
- /=dap-java-debug-*= cannot find a main class or test./ The project has
  not finished importing. Open a source file, wait for =LSP[jdtls]=, then
  retry.
- /Stale state after a dependency change./ =M-x lsp-workspace-restart=,
  or =C-c l R=, then re-launch the debug session. If the build file itself
  changed, use =C-c l j u= first to refresh JDT LS project configuration.
- /External file changes do not appear immediately./ Recursive LSP file
  watchers are disabled to keep Emacs memory bounded. Restart the workspace
  with =C-c l R= after large external changes.
- /Emacs memory still grows without bound./ Keep =lsp-log-io= off and close
  old project buffers so =lsp-keep-workspace-alive nil= can shut down idle
  servers. If memory still climbs, the next deeper fix is enabling
  =LSP_USE_PLISTS=true= before =lsp-mode= loads and recompiling/reinstalling
  =lsp-mode=.
- /No completions or diagnostics in a shell buffer./ The two binaries the
  shell stack relies on are not bundled with =lsp-mode= or Flycheck. Install
  with =npm install -g bash-language-server= and =brew install shellcheck=,
  then reopen the buffer (or =M-x lsp-workspace-restart=). =*lsp-log*= will
  name =bash-ls= explicitly when the language server is the missing piece.
- /No completions or diagnostics in a JS, TS, or TSX buffer./ Same shape as
  the shell case: =*lsp-log*= reports =ts-ls= not present (or "Command
  typescript-language-server ... not found"). Install with
  =npm install -g typescript-language-server typescript=, then reopen the
  buffer (or =M-x lsp-workspace-restart=). The project's own
  =node_modules/.bin/tsserver= does not count — that is the raw =tsserver=,
  not the LSP wrapper =ts-ls= launches.
- /ts-ls nags about non-issues (unreachable code, "declared but never
  read")./ Those are =tsserver= hint diagnostics, and the fix is a setting,
  not a different server. The ignore list is
  =systemhalted/lsp-ts-ignored-diagnostic-codes= in the Web editing section
  — a vector of numeric tsserver codes (7027, unreachable code, is already
  in it). Add the code from the offending Flycheck message, =C-c r= to
  reload the config, then =C-c l R= to restart the workspace. Note that
  =lsp-typescript-suggestion-actions-enabled= does /not/ silence these —
  =typescript-language-server= ignores that VS Code setting.

* LaTeX

Three layers: AUCTeX for editing and compilation, =auctex-latexmk= so
=latexmk= drives the build (handling cross-reference and bibliography
reruns automatically), and RefTeX for label / reference / citation
completion. The compiled PDF opens in the OS default viewer instead of
=pdf-tools= — fast to install, enough for the edit → compile → read →
save loop. Org's PDF export is rerouted through the same =latexmk=
command at the end of the section.

** Installation — TeX distribution

A working TeX Live providing =pdflatex= / =xelatex= / =latexmk= must be
on the shell =PATH= Emacs inherits (via =exec-path-from-shell=,
configured in the Programming section). The full distribution (~4 GB)
is preferred so missing-package chasing never blocks a build.

#+begin_src sh
# TeX Live (full distribution + latexmk):
# macOS, MacPorts:   sudo port install texlive +full
# macOS, Homebrew:   brew install --cask mactex-no-gui
# Fedora:            sudo dnf install texlive-scheme-full latexmk
# Ubuntu / Debian:   sudo apt install texlive-full latexmk

# Optional later: pdf-tools for in-Emacs PDF viewing with SyncTeX.
# macOS, MacPorts:   sudo port install poppler automake
# macOS, Homebrew:   brew install poppler automake
# Fedora:            sudo dnf install poppler-glib-devel automake
# Ubuntu / Debian:   sudo apt install libpoppler-glib-dev automake
# Then in Emacs: M-x pdf-tools-install
#+end_src

On macOS the installer adds =/Library/TeX/texbin= to =/etc/paths.d/=,
which login shells pick up automatically. On Linux the distro package
installs into =/usr/bin= so it's already on =PATH= without extra setup.
If =M-x executable-find RET latexmk= returns =nil= after installing,
restart the shell (or Emacs) so the new =PATH= entry is visible.

** AUCTeX — editing and compilation

[[https://www.gnu.org/software/auctex/][AUCTeX]] replaces the built-in
=latex-mode= with =LaTeX-mode=, a richer environment that knows about
TeX command structure, environments, sections, and math. The mode hooks
turn on three things on every LaTeX buffer:

- =turn-on-reftex= activates RefTeX (configured separately further down)
  for label / cite / ref completion.
- =LaTeX-math-mode= adds the ` (backtick) math-insert prefix —
  =` a= → =\alpha=, =` s= → =\sigma=, etc.
- =TeX-source-correlate-mode= enables SyncTeX so source ↔ PDF location
  jumps work (=TeX-source-correlate-start-server t= leaves the SyncTeX
  server running so viewers can call back into Emacs).

The custom settings:

- =TeX-auto-save t= and =TeX-parse-self t= cache parsed buffer
  structure across edits so completion stays responsive.
- =TeX-master nil= asks AUCTeX to prompt for the master file (the file
  that =\input='s the others) the first time per directory rather than
  guessing — wrong here means rebuilds against the wrong root.
- =TeX-PDF-mode t= compiles to PDF, not DVI; the mode-line shows
  =LaTeX/P= when it's active.
- =TeX-command-default "LatexMk"= makes =C-c C-c= default to the
  =LatexMk= command (registered by =auctex-latexmk= in the next
  subsection) instead of plain =pdflatex=.

The =:config= block sets up the PDF viewer. On macOS, =open -a
Preview.app= launches Preview; on Linux/BSD, =xdg-open= dispatches to
whatever the desktop has registered for =application/pdf= (Evince on
GNOME, Okular on KDE, Atril on MATE, etc.). Switching to =pdf-tools=
later is a matter of installing it (see the shell block above) and
changing this selection to ='((output-pdf "PDF Tools"))=.

#+begin_src emacs-lisp
(use-package tex
  :ensure auctex
  :hook
  ((LaTeX-mode . turn-on-reftex)
   (LaTeX-mode . LaTeX-math-mode)
   (LaTeX-mode . TeX-source-correlate-mode))
  :custom
  (TeX-auto-save t)
  (TeX-parse-self t)
  (TeX-master nil)
  (TeX-PDF-mode t)
  (TeX-source-correlate-method 'synctex)
  (TeX-source-correlate-start-server t)
  ;; latexmk handles reruns for refs/bib/TOC; auctex-latexmk registers it.
  (TeX-command-default "LatexMk")
  :config
  ;; Open the compiled PDF in the OS's default viewer.
  ;; macOS:        `open -a Preview.app' launches Preview.
  ;; Linux/BSD:    `xdg-open' dispatches to Evince/Okular/Atril/etc.
  ;; To switch to in-Emacs viewing later, install pdf-tools (see shell
  ;; block above) and change the selection to '((output-pdf "PDF Tools")).
  (if (eq system-type 'darwin)
      (setq TeX-view-program-selection '((output-pdf "Preview"))
            TeX-view-program-list      '(("Preview" "open -a Preview.app %o")))
    (setq TeX-view-program-selection '((output-pdf "xdg-open"))
          TeX-view-program-list      '(("xdg-open" "xdg-open %o")))))
#+end_src

** AUCTeX-latexmk — robust build driver

[[https://github.com/tom-tan/auctex-latexmk][auctex-latexmk]] registers a
=LatexMk= command with AUCTeX so =C-c C-c → LatexMk → RET= invokes
=latexmk= instead of stopping after the first =pdflatex= pass. =latexmk=
handles the reruns for cross-references (=\ref=, =\pageref=), citations
(=\cite=, =\bibliography=), and table-of-contents updates automatically;
without it, the user has to press =C-c C-c= repeatedly until references
stop changing. =auctex-latexmk-setup= wires the command into AUCTeX's
command list at load time.

#+begin_src emacs-lisp
(use-package auctex-latexmk
  :ensure t
  :after tex
  :config
  (auctex-latexmk-setup))
#+end_src

** RefTeX — labels, references, citations

RefTeX is built into Emacs and ships with the AUCTeX integration glue
already written; only two settings are needed.
=reftex-plug-into-AUCTeX t= tells RefTeX to take over =\ref=, =\cite=,
and =\label= insertion inside AUCTeX buffers — the picker UIs that the
LaTeX tutorial demonstrates with =C-c (=, =C-c )=, =C-c [=, and
=C-c ==. =reftex-default-bibliography nil= leaves the bibliography
list empty so RefTeX reads it from the =\bibliography{...}= directive
in each document rather than from a globally-configured file —
correct because different =.tex= files use different =.bib= sources.

#+begin_src emacs-lisp
(use-package reftex
  :ensure nil
  :custom
  (reftex-plug-into-AUCTeX t)
  (reftex-default-bibliography nil))
#+end_src

** Org PDF export via latexmk

The Org exporter's default PDF process is a hard-coded
=pdflatex → bibtex → pdflatex × 2= chain that doesn't always
converge on references or citations. Replacing
=org-latex-pdf-process= with a single =latexmk= invocation lets
=C-c C-e l p= in any =.org= file get the same correctness guarantees
=.tex= files already have. =-pdf= forces PDF output;
=-interaction=nonstopmode= prevents prompts from blocking the build;
=-output-directory=%o %f= keeps Org's chosen output directory.

#+begin_src emacs-lisp
;; Route Org's PDF export through latexmk so cross-refs and citations
;; resolve in one C-c C-e l p, matching the AUCTeX build above.
(with-eval-after-load 'ox-latex
  (setq org-latex-pdf-process
        '("latexmk -pdf -interaction=nonstopmode -output-directory=%o %f")))
#+end_src

** Tutorial: Using LaTeX

This tutorial is reachable from anywhere with =C-h T= -> /LaTeX/. It
covers the full edit -> preview -> PDF loop assuming TeX Live and the
Emacs blocks above are already installed.

Open or create a =.tex= file. The mode line should show =LaTeX/P= ---
that is AUCTeX's =LaTeX-mode= with =TeX-PDF-mode= active. If it shows
plain =LaTeX= without the =/P=, =TeX-PDF-mode= is off and builds will
target DVI; check the =:custom= block above.

Day-to-day workflow inside a =.tex= buffer:

| Keys              | Command                     | What it does                                |
|-------------------+-----------------------------+---------------------------------------------|
| =C-c C-c=         | =TeX-command-master=        | run the default command (LatexMk -> PDF)    |
| =C-c C-a=         | =TeX-command-run-all=       | build, then view, in one step               |
| =C-c C-v=         | =TeX-view=                  | open the compiled PDF (forward SyncTeX)     |
| =C-c C-p C-p=     | =preview-at-point=          | inline-render the math/figure at point      |
| =C-c C-p C-d=     | =preview-document=          | inline-render every preview in the file     |
| =C-c C-p C-c C-p= | =preview-clearout-at-point= | remove the inline preview at point          |
| =C-c (=           | =reftex-label=              | insert a label                              |
| =C-c )=           | =reftex-reference=          | pick from existing labels and insert =\ref= |
| =C-c [=           | =reftex-citation=           | pick from .bib entries and insert =\cite=   |
| =C-c ==           | =reftex-toc=                | navigable table of contents                 |

*Producing a PDF.* Press =C-c C-c=. AUCTeX prompts with =LatexMk= as
the default command --- accept with =RET=. The =*compilation*= window
shows latexmk's output; on success, =C-c C-v= opens the resulting PDF
in the OS's default viewer (Preview.app on macOS; whatever GNOME / KDE
/ etc. has registered for =application/pdf= on Linux --- typically
Evince, Okular, or Atril). The PDF lives next to the =.tex= source (or
under the directory set in a =% !TeX output-directory= magic comment,
if any). Save-a-copy keys differ per app: =Cmd-S= in Preview, =Ctrl-S=
in Evince / Okular / Atril --- that is "downloading" the PDF.

*Inline previews of math.* Place point inside any math environment
(e.g., between the =$= delimiters of =$x^2 + y^2 = z^2$=) and press
=C-c C-p C-p=. The TeX source is replaced visually by a rendered
image; the source restores itself the moment you edit the underlying
TeX. =C-c C-p C-d= renders every previewable region in the file ---
useful for skimming a long document. =C-c C-p C-c C-p= removes the
preview at point and goes back to source view explicitly.

*Cross-references and citations via RefTeX.* =C-c (= inserts a label
at the current location (RefTeX picks a sensible prefix like =eq:= or
=sec:= based on context). =C-c )= opens a picker over every label in
the document and inserts the matching =\ref{...}=. =C-c [= opens a
picker over the entries in any =\bibliography= file and inserts
=\cite{...}=. =C-c == shows a navigable table of contents in a side
buffer.

*Org -> PDF.* Inside any =.org= file, =C-c C-e l p= exports to PDF
using the =org-latex-pdf-process= override above. Same latexmk
invocation as =.tex= builds, so =\ref= / =\cite= / TOC reruns happen
in one shot.

*** Common failure modes

- /=C-c C-c= says "Searching for program... No such file or directory,
  latexmk"./ The TeX distribution is not on Emacs's =PATH=. Confirm
  with =M-x executable-find RET latexmk=. If =nil=, install per the
  shell block at the top of this section and restart Emacs so
  =exec-path-from-shell= picks up the new =bin= directory
  (=/Library/TeX/texbin= on macOS, =/usr/bin= on Linux when installed
  via the distro package).
- /Compile fails with "File `foo.sty' not found"./ Unlikely with
  =texlive +full= / =texlive-scheme-full= / =texlive-full=, but if it
  happens, install the specific collection the LaTeX error names
  (=sudo port install texlive-<collection>= on macOS / MacPorts;
  =sudo dnf install texlive-<collection>= on Fedora; =sudo apt install
  texlive-<collection>= on Ubuntu).
- /SyncTeX forward-search opens the PDF but does not jump to a
  location./ Expected with system viewers --- Preview.app and most
  Linux PDF readers (Evince, Atril, Okular under default settings)
  don't honour SyncTeX forward-search reliably. Install =pdf-tools=
  and switch =TeX-view-program-selection= to =PDF Tools= for true
  bidirectional SyncTeX.
- /Inline preview =C-c C-p C-p= renders nothing or a blank box./
  =preview-latex= shells out to =dvipng= or =gs=. Both ship inside
  =texlive +full=; if previews come up empty, check
  =M-x executable-find RET gs=.
